## 1. 文档说明

### 1.1 文档定位
本文件不是对 PRD 2.0 的重写，而是将 PRD 2.0 中已经确认的产品边界、目标和约束，进一步下钻为可执行的实施级设计文档，用于支撑研发、联调、测试、上线准备与后续 review。

### 1.2 目标
在不改变现有 MinerU、FastGPT、vLLM、桥接系统、飞书入口整体架构的前提下，补齐以下实施设计：

1. 飞书问答结果中的引用后处理机制。
2. 原始 PDF、Markdown 入库文档、FastGPT 引用结果之间的映射机制。
3. 单文件受控访问链路。
4. 权限校验边界与执行位置。
5. 最小审计、异常修复、监控与回滚要点。

### 1.3 范围边界
本文件仅覆盖 PRD 2.0 已确认范围内的实施设计，不扩展以下能力：

- 不开放目录浏览或文件树浏览。
- 不暴露真实文件路径、共享盘路径、NAS 路径。
- 不将原始 PDF 托管到飞书云作为正式方案。
- 不重构 FastGPT、MinerU、vLLM 的底层架构。
- 不扩展到页码级、段落级精准定位。
- 不扩展到 Office、图片、视频等更多原文件类型。

## 2. 设计输入分层：已确认事实 / 设计建议 / 待验证项

### 2.1 已确认事实
以下内容直接承接 PRD 2.0，可作为本实施文档的事实基础：

1. 当前链路为：原始 PDF → MinerU 转 Markdown → Markdown 入 FastGPT → FastGPT 问答 → 飞书返回结果。
2. 飞书是唯一用户入口，用户不直接访问 FastGPT 页面。
3. FastGPT、vLLM、内网知识服务部署在本地内网服务器。
4. 桥接系统部署于 Windows 工作电脑，负责连接飞书公网与内网服务，并承担跨子公司访问能力。
5. 本期核心目标是：在飞书中提供可靠、受控、可打开整份原始 PDF 的链接。
6. 本期必须避免：目录开放、真实路径暴露、原始 PDF 上云到飞书、模型自由生成真实链接。
7. 本期优先低风险、快速上线，不做大规模重构。

### 2.2 设计建议
以下内容为本实施文档给出的推荐设计，用于推动开发落地：

1. 在桥接系统增加“引用后处理 + 受控访问入口 + 审计埋点”能力。
2. 增加一张最小映射表，维护 PDF、Markdown、FastGPT 引用标识之间的关联。
3. 采用“短时效链接/令牌 + 服务端二次鉴权 + 单文件流式返回”的访问模式。
4. 将“打开原 PDF”动作的权限校验独立于问答结果。
5. 将映射异常、权限拒绝、链接过期、文件读取失败纳入统一错误分类与审计体系。

### 2.3 待验证项
以下事项在当前上下文中尚未被确认为既定事实，必须在开发推进中保留待验证口径：

1. **FastGPT 稳定可返回哪些引用字段**，以及不同问答模式下是否一致。
2. **权限基线来源**最终以哪个业务权限源、知识管理规则或最小配置方案为准。
3. **飞书身份到内部权限主体映射方式**是否已打通、能否稳定用于权限校验。
4. 桥接系统当前是否已具备签名链接、短时令牌、流式文件代理等能力，或需要新增。
5. 跨子公司访问场景下，组织标识是否统一、权限判断口径是否一致。
6. 原始 PDF 当前存储位置、版本管理和责任归属是否已梳理完备。

## 3. 实施目标与最小上线口径

### 3.1 本期最小上线目标（MVP）
本期最小上线版本以“飞书引用可信补链 + 原始 PDF 受控打开闭环”打通为目标，至少包含：

1. 飞书回答中展示来源，并对可识别来源生成“打开原 PDF”入口。
2. 系统能通过映射关系定位到单个原始 PDF。
3. 用户点击后，系统对打开动作执行身份识别、权限校验与审计记录。
4. 校验通过后，仅返回单个 PDF 的预览或下载流，不开放目录。
5. 映射缺失、权限不足、链接过期等异常有统一处理与留痕。

### 3.2 非目标提醒
以下能力即使后续可扩展，也不应被写成本期既定范围：

- 页码跳转
- 段落高亮
- 自动化全量历史映射补齐
- 全量统一权限平台重构
- 面向公网的开放下载系统

## 4. 一版主链路说明

### 4.1 文档入库建链主链路（文字时序）
1. 业务侧提供原始 PDF。
2. MinerU 将原始 PDF 转为 Markdown。
3. Markdown 以现有方式进入 FastGPT。
4. 入库补链流程回填或采集 FastGPT 可返回的候选引用标识。
5. 系统写入映射记录：原始 PDF ↔ Markdown 文档 ↔ FastGPT 引用候选字段。
6. 为该文档登记权限基线引用或最小访问范围配置。
7. 映射状态切换为“可用”，后续飞书问答方可生成原 PDF 打开入口。

### 4.2 飞书问答补链主链路（文字时序）
1. 用户在飞书发起提问。
2. 请求进入桥接系统。
3. 桥接系统调用 FastGPT 获取答案与引用结果。
4. 桥接系统提取引用结果中的候选标识。
5. 桥接系统查询映射表，尝试定位对应原始 PDF。
6. 若映射命中，则为该引用生成受控访问入口。
7. 桥接系统将“答案 + 引用来源 + 打开原 PDF”返回飞书。
8. 若映射未命中，则返回“来源可见但原文链接待补齐”的降级结果。

### 4.3 原始 PDF 打开主链路（文字时序）
1. 用户点击飞书中的“打开原 PDF”。
2. 请求到达桥接系统或桥接系统控制的受限文件访问能力。
3. 服务端校验链接签名、时效、目标文件标识是否合法。
4. 服务端识别当前用户身份。
5. 服务端根据文档权限基线执行权限校验。
6. 服务端写入审计日志。
7. 若校验通过，则按单文件方式返回 PDF 预览流或下载流。
8. 若校验失败，则返回统一错误提示，不泄露真实路径、文件位置或目录结构。

### 4.4 跨子公司访问链路说明
已确认事实是：跨子公司访问能力由桥接系统承担；因此本期访问链路设计必须统一经过桥接系统公网入口。

当前**未确认**的是：

1. 飞书身份是否可直接映射为集团统一的内部权限主体。
2. 子公司身份标识与总部权限体系是否统一。
3. 跨子公司共享文件的权限判断口径是否已统一。

因此本期设计结论是：

- **已确认事实**：跨子公司用户访问原始 PDF 时，必须统一走桥接系统出口，不应新增独立内网入口。
- **设计建议**：桥接系统在访问动作上统一完成身份识别、权限校验与审计留痕。
- **待验证项**：飞书身份到内部权限主体的映射方式，以及其对最终权限判断的影响边界，必须在开发前单独验证；在未验证前，不应写成“飞书身份天然等于可授权主体”。

### 6.4 唯一性与约束说明

#### 已确认事实
1. 不应将真实文件路径暴露给用户。
2. 不允许通过目录浏览方式访问文件。

#### 设计建议
1. 不以“文件名文本”作为唯一可信主键。
2. 映射定位优先采用稳定 ID；若当前阶段缺失稳定 ID，可使用“来源名称 + 数据集范围 + 版本 + 人工确认”的过渡策略。
3. 对同名、多版本文件必须可区分。
4. storage_ref 仅供服务端定位，不对外暴露。

#### 待验证项
1. FastGPT 哪些字段可被作为稳定候选主键。
2. 历史文档是否已具备足够一致的命名与版本规范。

## 8. 安全 / 权限 / 审计控制点清单

### 8.1 安全控制点
1. 链接必须由系统生成，不允许模型拼接真实地址。
2. 链接需为短时效且带签名/令牌校验能力。
3. 访问目标只能是单个 PDF，不允许目录级浏览。
4. 不在任何对外响应中暴露真实文件系统路径、共享盘地址、NAS 地址。
5. 不允许通过修改 URL 参数枚举其他文件。
6. 打开失败时，不返回文件是否存在、路径是否有效等内部信息。
7. 预览与下载的底层控制逻辑应一致，不应出现一种方式有鉴权、另一种方式无鉴权。

### 8.2 权限控制点
1. 问答命中不等于原文可打开。
2. “打开原 PDF”动作必须独立校验。
3. 权限校验至少依赖：当前访问主体、文档权限基线引用、业务范围信息。
4. 若存在多层规则，应以更严格规则优先。
5. 权限不足时，拒绝访问并记录原因码。
6. 跨子公司访问不得自动放宽权限边界。

### 8.3 审计控制点
1. 链接生成建议记录审计事件。
2. 打开成功必须记录审计事件。
3. 打开失败必须记录审计事件。
4. 映射缺失、映射冲突、权限拒绝、链接过期均需有明确分类码。
5. 审计日志查询权限必须受控，仅管理员/安全角色可见。

## 10. 映射异常状态与修复机制

### 10.1 推荐映射状态

| 状态 | 含义 | 是否可对外补链 |
| --- | --- | --- |
| draft | 已登记但未完成校验 | 否 |
| active | 映射有效 | 是 |
| disabled | 已下线或停用 | 否 |
| error | 映射异常待修复 | 否 |

### 10.2 典型异常场景
1. FastGPT 返回字段无法命中映射。
2. 同名文件或多版本文件导致命中冲突。
3. 原始 PDF 已移动或失效，storage_ref 不可用。
4. 文档已下线但映射未停用。
5. 历史文档缺少权限基线配置。

### 10.3 修复机制建议
1. 对映射查询失败，记录错误码并输出“原文链接待补齐”。
2. 对映射冲突，禁止自动补链，转为人工确认修复。
3. 对 storage_ref 失效，标记状态为 error，暂停对外补链。
4. 对权限基线缺失，禁止文件打开并记录配置缺失问题。
5. 修复动作必须记录操作人、修复前值、修复后值、修复原因。

### 10.4 待验证项影响说明
- **FastGPT 引用字段**未确认前，自动映射命中逻辑不能写死；实施时应先做样本验证。
- **权限基线来源**未确认前，权限校验模块应支持最小配置兜底，但不得默认放行。
- **飞书身份到内部权限主体映射**未确认前，跨子公司与敏感文档场景应视为高风险校验点。

## 11. 基本运维监控与回滚要点

### 11.1 基本监控指标
建议至少监控以下指标：

1. 飞书问答补链成功率。
2. 映射命中率。
3. 原 PDF 打开成功率。
4. 权限拒绝率。
5. 链接过期失败率。
6. 文件读取失败率。
7. 平均打开耗时与 P95 耗时。
8. 跨子公司访问成功率。

### 11.2 基本告警建议
1. 映射命中率显著下降。
2. 文件打开失败率持续升高。
3. 权限校验异常全量失败或全量放行。
4. 桥接系统文件流式返回异常。
5. 同一文件出现异常高频访问或异常高频拒绝。

### 11.3 回滚要点
本期能力应支持以下最小回滚策略：

1. **补链回滚**：关闭“打开原 PDF”入口，仅保留问答与来源展示，不影响主问答链路。
2. **访问回滚**：关闭受控文件访问能力，对外统一返回“暂不可打开原文”。
3. **映射回滚**：停用问题映射记录，避免错误文件被打开。
4. **权限回滚**：当权限校验规则异常时，优先回退到“拒绝打开”而非“默认放行”。

### 11.4 回滚原则
1. 回滚优先保证安全，不优先保证功能暴露。
2. 问答主链路与原文打开链路应解耦，避免文件访问能力故障拖垮问答主链路。
3. 对用户侧提示应统一，避免暴露内部故障细节。

## 12. 跨子公司访问链路、身份映射与权限边界

### 12.1 已确认事实
1. 桥接系统已承担跨子公司访问能力。
2. 原始 PDF 打开能力本期必须纳入桥接出口统一处理。
3. 用户唯一入口仍是飞书。

### 12.2 设计建议
1. 飞书点击打开原 PDF 后，请求统一进入桥接系统公网入口。
2. 桥接系统负责识别当前飞书用户上下文，并转换为权限校验所需的访问主体。
3. 若后续有内部统一身份主体体系，应由桥接系统在服务端完成映射，不在前端暴露任何组织、路径或规则细节。
4. 对跨子公司访问，应在文档权限校验中同时考虑 business_scope 与组织归属。

### 12.3 待验证项
以下内容当前不能写成既定事实：

1. 飞书身份字段是否可稳定映射到内部员工唯一主体。
2. 集团与子公司之间是否已有统一权限主体体系。
3. 若飞书组织身份与内部权限主体存在一对多或多对一情况，应如何处理。
4. 对跨子公司共享文件，最终是按集团统一授权还是按子公司独立授权。

### 12.4 对权限校验的影响边界
1. 在**身份映射未确认**前，权限校验模块只能设计为“预留映射接口 + 最小可配置兜底”，不能假设飞书用户 ID 就是最终授权主体。
2. 在**跨子公司权限边界未确认**前，不应默认允许子公司用户访问总部文档或反向访问。
3. 在**权限基线来源未确认**前，系统只能对最小配置范围内的文档开放受控访问，不得默认所有问答命中文档都可打开。

本设计开发文档承接 PRD 2.0，将“飞书引用来源补链 + 原始 PDF 受控打开”进一步落到实施层，核心落点是：
