- 文档名称：公司内部知识库 PRD 初版
- 文档版本：v0.1
- 文档状态：review 已通过，待正式发布
- 适用阶段：产品需求定义
- 输出日期：2026-03-30
- 适用读者：产品、研发、管理层、信息安全/IT、合规相关角色

## 1. 文档概述

### 1.1 项目背景
公司现有知识内容分散在即时通讯、个人文档、共享盘、邮件附件与临时文件中，存在以下问题：

1. 知识沉淀分散，难以形成统一入口。
2. 关键制度、流程、项目经验缺乏版本管理与责任归属。
3. 异地办公场景增加后，内部知识访问需求提升，但访问安全控制能力不足。
4. 文档权限边界不清，存在误看、误传、越权导出风险。
5. 关键操作缺少完整留痕，不利于审计、复盘与责任追踪。

因此，需要建设一套面向企业内部的知识库产品，用于支持知识统一存储、权限化访问、受控协作、审计追溯与安全管理。

### 1.2 建设目标
本项目目标是建设一套公司内部知识库系统，满足以下核心诉求：

1. 建立统一知识沉淀与检索入口。
2. 支持文档从创建、审核、发布、归档到废弃的完整生命周期管理。
3. 支持基于组织角色、知识空间、文档分级的权限分级控制。
4. 支持异地办公场景下的安全登录与受控访问。
5. 支持关键业务动作的可溯源与审计。
6. 在初版阶段明确产品边界、控制原则和实施前提，为后续原型、技术方案与项目排期提供依据。

### 1.3 产品定位
本产品定位为**企业内部知识资产管理平台**，重点解决“内部知识集中管理、分级访问、受控协作、可追溯审计”问题。

本轮为 PRD 初版，重点是产品需求定义，不包含：

- 原型图与交互细节设计
- 技术架构与部署方案定板
- 数据库模型与接口文档
- 外部协作平台的深度打通细节
- 全量历史知识迁移实施方案

### 1.4 适用范围
本 PRD 适用于：

- 企业内部员工知识使用场景
- 部门级、项目级、制度级知识沉淀场景
- 内部受限访客访问场景
- 知识审核、发布、追溯、审计场景

### 1.5 非适用范围
本期不纳入以下范围：

- 外部匿名访问
- 面向公众的公开知识门户
- 外部合作方开放注册
- 社区化内容运营
- 复杂内容创作工作台能力
- 详细技术实现与供应商选型结论

### 1.6 术语定义
- **知识空间**：按部门、业务线、项目或专题划分的知识容器。
- **知识条目**：知识库内的页面、文档、制度说明、操作手册等内容对象。
- **文档分级**：按内容敏感度定义的访问控制等级，本文建议分为普通、内部、敏感、受限四档。
- **异地登录**：员工在非公司办公网络、异地办公、出差、远程办公等场景下访问知识库的登录行为。
- **可溯源**：对创建、修改、审批、发布、访问、导出、权限变更等关键动作进行记录，并支持按对象、人员、时间追查。
- **访客**：默认指**内部受限访客**，如短期借调人员、实习生、跨部门只读访问人员；不包含外部匿名用户，也不默认包含外部合作伙伴。

## 2. 业务价值与问题定义

### 2.1 当前痛点
1. 知识内容分散，重复咨询成本高。
2. 经验文档依赖个人维护，人员流动时知识易丢失。
3. 缺少统一版本与审批机制，制度类文档可信度不足。
4. 搜索结果质量不稳定，且常出现搜到但无权限访问的体验问题。
5. 高敏感内容缺少导出、异地访问、权限变更的控制与审计。

### 2.2 业务价值
1. 提升组织知识复用效率，减少重复沟通与重复建设。
2. 将知识资产从“个人占有”转为“组织可管理资产”。
3. 降低员工异地办公下的知识访问阻力，同时保留安全控制能力。
4. 为管理层提供制度发布、经验沉淀、责任追踪的基础能力。
5. 为后续合规、内控、安全审计提供记录基础。

### 2.3 初版成功标准方向
本轮 PRD 不承诺运营指标数值，但产品目标应支撑以下方向：

- 核心知识可被统一沉淀与检索。
- 文档状态流转清晰，发布内容可信。
- 权限边界明确，越权暴露风险受控。
- 异地登录具备风险控制能力。
- 关键操作可审计、可追溯。

## 3. 用户角色与职责

## 3.1 角色定义

| 角色 | 定义 | 典型职责 |
| --- | --- | --- |
| 访客 | 内部受限访客，仅授予必要范围的只读或受限评论权限 | 查看指定空间/文档 |
| 员工 | 普通内部员工 | 查看、检索、创建个人或授权范围内内容 |
| 编辑者 | 承担知识维护职责的业务角色 | 编辑、提交审核、维护分类与内容质量 |
| 审批人 | 对制度、规范、关键知识负审批责任的角色 | 审核、驳回、发布审批 |
| 空间管理员 | 负责某一知识空间的权限与内容治理 | 配置空间成员、分类、部分审计查看 |
| 知识管理员 | 负责全局知识治理与平台规范 | 模板、分级规则、全局配置、归档治理 |
| 系统管理员 | 负责系统配置与运维接入的 IT 角色 | 身份接入、日志策略、系统配置、故障处理 |
| 安全/审计角色 | 负责异常行为核查、取证与合规检查 | 审计日志查询、风险核查、导出复核 |

### 3.2 角色口径说明
1. “访客”仅指**内部受限访客**，默认不开放为外部匿名访问角色。
2. 外部合作方、供应商、客户访问不在本期默认范围内；若后续要支持，需单列产品方案与安全评审。
3. 系统管理员不自动等同于所有文档内容可读，涉及高敏感内容时仍应遵循最小权限原则与审计要求。

## 4. 产品范围

### 4.1 本期范围
本期聚焦以下能力：

1. 账号登录与身份识别
2. 异地登录安全控制能力定义
3. 知识空间与分类管理
4. 文档创建、编辑、版本管理
5. 审核、发布、归档、废弃流转
6. 权限分级与分层控制
7. 搜索与检索最小能力
8. 关键操作日志与审计追溯
9. 基础导出控制与导出留痕

### 4.2 非本期范围
1. 原型设计与高保真交互
2. 复杂工作流引擎设计
3. OCR、语义检索、智能问答等增强能力
4. 公网公开链接分享
5. 外部匿名访问与开放注册
6. 多租户 SaaS 商业化能力
7. 大规模历史知识自动迁移方案
8. 细化技术架构、接口与数据库设计

### 4.3 历史知识迁移边界
默认建议：

- 本期以新建知识与人工整理导入为主。
- 支持有限度的手工上传附件或拷贝录入。
- 批量导入、复杂结构化迁移、跨系统自动同步列为后续范围或待确认项。

## 5. 核心业务目标与需求原则

### 5.1 业务目标
1. 形成统一知识入口与基础治理机制。
2. 保证关键知识内容可被授权访问、维护和复用。
3. 在远程/异地访问场景下提供必要安全控制。
4. 对关键内容与关键动作建立责任链与审计链。

### 5.2 需求原则
1. **内部优先**：以企业内部使用场景为唯一默认场景。
2. **最小权限**：默认按最小必要范围授予访问权。
3. **权限先于检索暴露**：搜索结果展示必须先过权限判断。
4. **留痕优先**：关键动作应默认进入追溯体系。
5. **状态清晰**：知识内容必须具备明确生命周期。
6. **前提透明**：依赖外部系统的能力必须显式标注，不得写成已内建。

## 6. 知识对象范围与生命周期

### 6.1 知识对象范围
本产品纳入统一管理的知识对象包括：

1. 页面/正文内容
2. 附件
3. 评论
4. 文档版本记录
5. 审批记录
6. 导出文件记录
7. 权限配置记录

### 6.2 生命周期状态定义
知识条目至少应支持以下状态：

| 状态 | 定义 | 是否可编辑 | 是否可被检索 | 说明 |
| --- | --- | --- | --- | --- |
| 草稿 | 作者编辑中，未进入正式评审 | 是 | 否/仅本人及授权协作者可见 | 默认不面向普通搜索公开 |
| 审核中 | 已提交审批，等待处理 | 受限编辑 | 受限 | 审批中不允许无痕修改 |
| 已发布 | 通过审核并对授权范围可见 | 通过变更流程编辑 | 是 | 作为正式内容提供访问 |
| 已归档 | 不再主动维护，但需保留历史价值 | 否/仅管理员可维护 | 可按策略检索 | 默认弱化展示 |
| 已废弃 | 已不再适用，不推荐继续使用 | 否 | 默认不在普通搜索结果中前置展示 | 允许保留追溯记录 |

### 6.3 生命周期规则
1. 草稿可由作者或协作者编辑。
2. 审核中的内容应冻结主要正文；如需修改，应退回草稿或保留变更记录。
3. 已发布内容的更新应形成新版本，并保留历史版本可追溯。
4. 已归档内容可被授权查看，但应标识为历史资料。
5. 已废弃内容不得作为默认推荐版本，但须保留来源与废弃原因。

### 6.4 生命周期责任归属
1. 内容创建人负责草稿阶段初始录入。
2. 编辑者或空间管理员负责内容维护质量。
3. 审批人负责关键文档发布前审核。
4. 知识管理员负责归档规则与废弃治理。

## 7. 账号、登录与身份管理

### 7.1 账号体系要求
1. 系统应基于企业内部账号体系进行身份识别。
2. 可支持邮箱、工号或统一身份标识作为登录凭据入口，具体方式待后续确认。
3. 本期不支持外部匿名注册或公开注册。
4. 账号状态至少应包含：启用、停用、冻结。

### 7.2 账号与权限生命周期
为解决 review 指出的生命周期缺口，本期明确如下规则：

#### 7.2.1 入职
1. 新员工账号开通后，应按组织、岗位或默认角色授予基础访问权限。
2. 默认仅获得最小必要权限，不自动获得敏感空间访问权。
3. 若需特殊空间权限，应走权限申请或管理员授权流程。

#### 7.2.2 转岗/调岗
1. 员工组织归属或岗位变化后，应支持权限重新评估。
2. 原岗位不再需要的空间访问权限应回收或降级。
3. 新岗位所需权限应重新授权，不得沿用历史高权限作为默认。
4. 转岗引发的权限变更应纳入审计。

#### 7.2.3 离职/停用
1. 员工离职、长期停用、账号冻结后，应禁止继续登录与访问。
2. 与个人账号绑定的草稿、已发布文档、审批记录、操作日志不得因账号停用而丢失。
3. 已发布知识资产归属应转移至组织或指定维护人；原作者身份保留在历史记录中。
4. 离职后权限应完成回收，并保留相应审计记录。

#### 7.2.4 权限回收
1. 权限回收适用于离职、转岗、临时授权到期、项目结束、审计整改等场景。
2. 系统应支持按用户、按空间、按角色、按时效进行权限回收。
3. 回收动作应保留操作者、原因、时间、生效范围等审计字段。

#### 7.2.5 知识资产归属
1. 知识内容原则上归属于组织，不归属于个人私有资产。
2. 作者、协作者、审批人、发布人等责任信息应长期保留在版本历史中。
3. 员工离职后，其创建内容的维护权应可转移，但历史责任链不得被覆盖。

#### 7.2.6 审计保留
1. 账号生命周期相关动作（开通、停用、角色调整、权限回收）应进入审计日志。
2. 审计保留时长为本期必须定义的管理项；若企业具体时长未定，可在 PRD 中标为待确认，但不得留空。

### 7.3 登录控制基本要求
1. 支持正常办公网络与异地办公场景下的登录访问。
2. 登录后会话应具备超时、失效与重新认证机制。
3. 对异常登录、失败登录、频繁尝试应支持留痕与告警基础能力。

## 8. 异地登录与安全控制

### 8.1 能力目标
系统需要具备针对异地登录的安全控制能力，用于支持远程办公、出差、非公司网络访问等场景下的受控登录与访问。

### 8.2 口径统一说明
本 PRD 明确：

1. 产品**需要具备异地登录安全控制能力**。
2. 该能力可通过**产品自建能力**或**对接外部系统（如 SSO、MFA、VPN、零信任、企业安全网关）**实现。
3. 本轮 PRD **不默认这些能力已内建**，也不直接锁定具体供应商或技术方案。
4. 未确认接入方案前，不应在后续方案中将 MFA、VPN、零信任等写成已具备事实。

### 8.3 最小能力要求
1. 识别登录来源的基础风险信号，如地域、网络环境、设备状态或账号行为异常。
2. 对高风险异地登录支持额外校验、限制访问、要求审批或触发告警。
3. 对高敏感知识内容可配置更严格的异地访问策略。
4. 保留异地登录相关审计记录。

### 8.4 异地访问控制原则
按文档分级建议如下：

| 文档分级 | 异地访问默认策略 |
| --- | --- |
| 普通 | 允许登录后访问 |
| 内部 | 允许访问，必要时进行风险校验 |
| 敏感 | 允许在满足附加校验条件后访问 |
| 受限 | 默认禁止直接异地访问，或需额外审批/受控网络环境 |

### 8.5 异常处置要求
1. 对异常地区登录、异常设备登录、异常频次登录，应支持记录与告警。
2. 对高敏感内容的异常异地访问，应支持中断访问、要求补充验证或提交复核。
3. 安全策略触发后，应保留事件编号、触发规则、处置结果等记录。

## 9. 权限分级模型

### 9.1 分层模型
本系统采用三层权限模型：

1. **系统级权限**：决定用户是否可管理平台级配置、查看全局日志、维护全局规则。
2. **空间级权限**：决定用户在某一知识空间内的查看、编辑、管理权限。
3. **文档级权限**：决定用户对特定文档的查看、评论、编辑、审批、发布、导出等权限。

### 9.2 文档分级
文档分级建议如下：

| 分级 | 说明 | 最小控制原则 |
| --- | --- | --- |
| 普通 | 一般性工作知识 | 对授权员工开放查看，导出可允许并留痕 |
| 内部 | 公司内部常规资料 | 仅内部授权人员可看，不允许外部分享 |
| 敏感 | 涉及经营、客户、流程细节等敏感资料 | 需更严格权限与导出控制，异地访问需附加校验 |
| 受限 | 高敏感、重大风险资料 | 最小范围授权，导出受限，异地访问默认更严格控制 |

| 权限项 | 访客 | 员工 | 编辑者 | 审批人 | 空间管理员 | 知识管理员 | 系统管理员 | 安全/审计 |

| 查看授权内容 | 有限 | 是 | 是 | 是 | 是 | 是 | 受限 | 受限 |
| 搜索授权内容 | 有限 | 是 | 是 | 是 | 是 | 是 | 受限 | 受限 |
| 创建内容 | 否 | 有限 | 是 | 有限 | 是 | 是 | 否 | 否 |
| 编辑内容 | 否 | 有限 | 是 | 有限 | 是 | 是 | 否 | 否 |
| 评论内容 | 有限 | 是 | 是 | 是 | 是 | 是 | 否 | 否 |
| 提交审核 | 否 | 有限 | 是 | 有限 | 是 | 是 | 否 | 否 |
| 审批/驳回 | 否 | 否 | 否 | 是 | 有限 | 是 | 否 | 否 |

| 配置全局规则 | 否 | 否 | 否 | 否 | 否 | 有限 | 是 | 否 |
| 导出内容 | 否/有限 | 有限 | 有限 | 有限 | 有限 | 有限 | 否 | 有限 |
| 查看审计日志 | 否 | 否 | 否 | 否 | 有限 | 有限 | 有限 | 是 |
| 变更权限 | 否 | 否 | 否 | 否 | 有限 | 是 | 有限 | 否 |

说明：
1. “有限”表示需受空间、文档分级、审批流或额外授权限制。
2. 系统管理员默认不以“可读全部业务内容”为权限前提，涉及内容读取时应受额外审批或留痕约束。
3. 安全/审计角色主要访问日志与风险数据，不默认拥有业务内容编辑权限。

### 9.4 权限控制原则
1. 默认拒绝原则：未被授予即不可访问。
2. 权限继承需可见：空间权限与文档例外权限关系需明确展示。
3. 高敏感文档禁止以“组织内所有人可见”作为默认策略。
4. 导出、打印、复制等动作应纳入高风险操作控制范围。

## 10. 核心功能需求

### 10.1 知识空间与分类管理
1. 支持按部门、业务线、项目、专题建立知识空间。
2. 支持空间内目录、标签、分类管理。
3. 支持空间级成员配置与角色授权。
4. 支持空间级内容模板或规范提示。

### 10.2 文档创建与编辑
1. 支持新建文档、草稿保存、协同编辑基础能力。
2. 支持正文与附件管理。
3. 支持修改记录与版本对比基础能力。
4. 支持文档分级、归属空间、标签等基础元数据维护。

### 10.3 审核与发布
1. 关键文档需支持提交审核。
2. 审批人可执行通过、驳回、备注。
3. 已发布内容应带有版本号、发布时间、发布责任人信息。
4. 更新已发布内容时应形成新版本而非覆盖历史。

### 10.4 归档与废弃
1. 长期不维护或被替代内容应支持归档或废弃。
2. 被新版本替代的旧文档应可建立关联关系。
3. 已废弃内容应显著标识，不作为默认推荐内容。

### 10.5 导出与分享控制
1. 是否允许导出应受文档分级与权限限制。
2. 高敏感文档导出应支持审批或强留痕。
3. 本期不支持公开分享链接。
4. 内部分享应以受控链接或受控访问入口为主。

---

## 11. 搜索与检索边界

### 11.1 检索对象范围
本期搜索最小能力覆盖：

1. 标题
2. 正文
3. 文档标签/分类
4. 已授权可见的附件名称

以下能力不默认纳入本期：

- 附件全文检索
- 图片 OCR 检索
- 评论全文检索
- 跨系统统一搜索
- 语义检索与智能问答

### 11.2 权限过滤规则
1. 搜索前或搜索结果返回前必须进行权限过滤。
2. 对无权限文档，不得展示正文摘要、附件名称、敏感标签等可能形成信息泄露的内容。
3. 可根据需要返回“存在相关内容但无权限查看”的弱提示，但不得暴露内容详情。

### 11.3 结果展示边界
1. 搜索结果至少展示标题、所属空间、更新时间、发布状态等基础信息。
2. 对敏感与受限文档，结果展示应遵循最小暴露原则。
3. 草稿、审核中、归档、废弃内容的展示策略应按角色控制。

### 11.4 检索体验要求
1. 支持按空间、标签、状态、作者、更新时间等基础筛选。
2. 支持关键词检索与结果排序。
3. 搜索性能目标本期仅定义方向，不在 PRD 初版承诺硬指标。

---

## 12. 可溯源与审计要求

### 12.1 可溯源目标
系统应支持对知识内容与关键安全动作进行责任链追踪，至少满足“谁在什么时间对什么对象做了什么动作，结果如何”的记录要求。

### 12.2 需留痕的关键动作
至少包括：

1. 登录成功/失败
2. 异地登录与异常登录事件
3. 文档创建、编辑、删除、恢复
4. 版本发布、回滚、归档、废弃
5. 审批通过、驳回、撤回
6. 权限申请、授权、变更、回收
7. 敏感文档查看、导出、打印等高风险访问动作
8. 空间成员调整与分级调整

### 12.3 权限变更审计字段
针对 review 第三轮指出的缺口，本期明确权限变更日志至少包含以下字段：

| 字段 | 说明 |
| --- | --- |
| 事件ID | 权限变更事件唯一标识 |
| 变更时间 | 发生时间 |
| 操作者 | 执行变更的人员/系统 |
| 目标对象类型 | 用户、角色、空间、文档、策略 |
| 目标对象ID/名称 | 被变更对象标识 |
| 变更前权限 | 调整前权限状态 |
| 变更后权限 | 调整后权限状态 |
| 变更范围 | 系统级、空间级、文档级 |
| 变更原因 | 申请、审批、转岗、离职、整改等 |
| 审批单号/关联流程ID | 如通过审批流程触发，应记录关联单据 |
| 生效时间 | 权限生效时间 |
| 失效时间 | 临时权限需记录失效时间 |
| 来源渠道 | 手工操作、批量导入、系统同步、接口调用 |
| 处置结果 | 成功、失败、部分成功 |
| 备注 | 补充说明 |

### 12.4 内容版本追溯字段
文档版本记录至少包括：

- 文档ID
- 版本号
- 创建人/修改人
- 修改时间
- 变更摘要
- 审批状态
- 发布人/发布时间
- 回滚来源版本（如有）

### 12.5 审计日志访问控制
1. 普通员工不得查看全量审计日志。
2. 空间管理员仅可查看其管理范围内的必要日志。
3. 安全/审计角色可按授权查看安全事件与审计日志。
4. 审计日志本身的查看动作也应纳入留痕。

### 12.6 审计保留最小规则
1. 审计日志不得由普通员工删除或篡改。
2. 高敏感文档访问、导出、权限变更应纳入重点保留范围。
3. 审计保留周期为执行前必须确认的治理项；若企业未定，本期可写为候选值并标注待确认，但不得空缺。

---

## 13. 安全与审计最小规则

### 13.1 最小规则清单
1. 默认不支持公开分享链接、外部匿名访问、开放注册。
2. 用户、设备、登录位置、访问日志、导出日志等安全数据仅用于安全控制与审计目的。
3. 高敏感内容的查看、导出、权限变更、异地访问应纳入重点审计。
4. 普通员工不得删除、修改审计日志。
5. 搜索结果不得泄露无权限内容的正文摘要与敏感信息。
6. 高风险事件应支持告警或至少强留痕。
7. 账号停用后必须阻断访问，但相关知识资产与历史责任链必须保留。
8. 系统管理员的高权限操作也必须被记录与可审计。

### 13.2 最小化采集原则
1. 仅采集实现登录风控、访问审计、事件追查所需的最小数据。
2. 不得将安全数据用于与知识管理无关的扩展用途。
3. 对安全数据的访问应单独授权、单独留痕。

## 14. 关键业务流程

### 14.1 登录访问流程
1. 用户进入知识库登录入口。
2. 系统校验身份凭证或跳转至企业身份体系。
3. 系统评估登录风险（含是否异地、设备/环境异常等）。
4. 若命中高风险策略，则执行附加验证、限制访问或人工审批。
5. 登录成功后，按用户角色与权限范围展示知识空间与可访问内容。
6. 全流程写入登录与安全审计日志。

### 14.2 文档创建与发布流程
1. 作者创建文档并保存为草稿。
2. 作者填写分类、标签、分级、归属空间等元数据。
3. 编辑完成后提交审核。
4. 审批人执行通过或驳回。
5. 审核通过后发布为正式版本，系统记录版本号、发布时间、责任人。
6. 后续变更形成新版本并保留历史版本。

### 14.3 权限申请与审批流程
1. 用户申请访问某空间或敏感文档权限。
2. 系统记录申请原因、访问范围、时长需求。
3. 管理员或审批人审核申请。
4. 审核通过后授予权限，并记录生效时间、失效时间、审批单据。
5. 到期或条件变化后回收权限，并保留审计记录。

### 14.4 版本追溯流程
1. 用户查看文档版本历史。
2. 系统展示版本号、修改人、时间、变更摘要、审批状态。
3. 对已发布版本支持查看差异或回滚申请。
4. 回滚行为需产生新的版本记录与审计记录。

### 14.5 敏感文档导出审计流程
1. 用户发起敏感文档导出。
2. 系统按分级与权限判断是否允许、是否需审批。
3. 执行导出前记录申请信息与原因。
4. 导出完成后记录导出时间、执行人、对象、结果。
5. 对异常频次或异常场景导出触发告警或复核。

## 15. 非功能要求

### 15.1 安全性
1. 支持身份认证、会话控制、权限校验、关键操作留痕。
2. 支持对高风险登录与高敏感内容访问进行附加控制。
3. 敏感数据访问遵循最小权限原则。

### 15.2 稳定性
1. 作为企业内部基础系统，需具备稳定访问能力。
2. 关键日志记录机制不应因普通业务失败而完全缺失。

### 15.3 可用性
1. 支持员工快速找到授权范围内内容。
2. 关键流程（登录、搜索、查看、提交审核）应保持可理解与可操作。

### 15.4 性能
1. 本期不定义硬性性能 SLA，但应支持常规企业知识访问规模。
2. 搜索与打开文档体验应可接受，具体指标待技术方案阶段补充。

### 15.5 可审计性
1. 审计日志应支持按用户、文档、空间、时间、动作类型查询。
2. 审计日志导出与查看行为应受控。

## 16. 待确认项与处理规则

### 16.1 可写为选项、允许后续确认的事项
以下事项可在本 PRD 中以“候选方案/待确认项”记录，不阻断本轮 PRD 初版交付：

1. 登录标识采用邮箱、工号或二者兼容。
2. 身份认证接入方式采用 SSO、自建登录或混合方案。
3. MFA 是否对全员强制，还是仅针对高风险场景触发。
4. 是否接入 VPN、零信任或企业安全网关。
5. 是否支持移动端，以及支持 H5、App 或移动浏览器范围。
6. 审计保留周期候选值（如 180 天、1 年、2 年等）最终采用哪档。
7. 历史知识迁移是否纳入后续阶段。

### 16.2 执行前需明确、属于阻塞前提的事项
以下事项若进入设计或开发执行阶段前仍未明确，将构成实施阻塞：

1. 企业身份源与账号接入方式。
2. 权限分级口径与文档分级标准的最终确认。
3. 高敏感内容异地访问的最终控制策略。
4. 审计日志保留周期与查看授权边界。
5. 是否允许导出敏感/受限文档及对应审批要求。
6. 访客角色是否仅限内部受限访客的管理口径确认。

### 16.3 处理规则
1. 对不阻断本轮 PRD 的事项，可在文档中作为选项保留，供后续评审决策。
2. 对阻塞实施的前提，必须在进入原型、技术方案、开发排期前完成确认。
3. 未确认事项不得在后续文档中被表述为已落地能力。

## 17. 风险与约束

1. 若企业已有复杂身份/安全体系，本产品边界需与既有系统明确切分，否则职责重叠。
2. 权限模型若过细，初版可能实施复杂度过高；需平衡治理精度与落地成本。
3. 审计与安全规则若无管理制度配套，可能出现“有记录但无人处置”的执行落差。
4. 历史知识迁移若强行纳入本期，可能影响初版交付节奏与范围控制。

## 18. 验收标准

本 PRD 初版完成并可进入评审，至少满足以下标准：

1. 结构完整，适合产品、研发、管理层共同评审。
2. 明确覆盖异地登录、权限分级、可溯源、知识生命周期、搜索与检索边界、安全与审计最小规则。
3. 明确给出角色定义、角色-权限矩阵与文档分级原则。
4. 明确给出账号与权限生命周期规则。
5. 明确澄清“访客”角色口径仅限内部受限访客。
6. 明确统一异地登录能力口径：可自建或集成实现，但本轮不默认已内建。
7. 明确列出权限变更审计字段。
8. 明确待确认项的处理规则，并区分可选项与执行阻塞前提。

10. 不扩展到原型、接口、技术架构细化。
