## 1. 文档边界与证据标签

## 1.1 这份 SOP 管什么，不管什么

**这份文档负责：**
- 规定团队如何试点、执行、review、验收、沉淀 Codex 使用方式
- 给出主干流程、角色分层、风险边界、落地 checklist
- 作为 GitHub / Notion 可直接发布的内部标准件

**这份文档不负责：**
- 重复介绍 Codex 所有入口与能力细节
- 代替 `codex-guide.md` 成为知识总览
- 代替 `sources.md` 成为来源仓库
- 代替 `codex-prompt-templates.md` 成为模板全集

如需背景认知，请回看：
- 使用指南：[`codex-guide.md`](./codex-guide.md)
- 提示词模板：[`codex-prompt-templates.md`](./codex-prompt-templates.md)
- 来源清单：[`sources.md`](./sources.md)

## 1.2 证据标签规则

为避免把新增流程建议伪装成官方事实，本文档显式区分：

- **【直接沿用已有材料的结论】**：可直接回到已有 guide / sources 找依据
- **【基于已有材料的归纳整理】**：由已有材料抽象出的组织与执行方法
- **【新增实践建议 / 待复核判断】**：适合内部采用，但不应误写成 OpenAI 官方硬规则

培训、发布、复用时，默认优先保留原标签，不做无标注转述。

## 2. 主干流程：团队最小可执行 SOP

> 目标：让团队在低风险范围内，形成“任务定义 → plan → execute → verify → review → 沉淀”的闭环。

## 2.1 阶段 0：试点前准备

**【基于已有材料的归纳整理】**
必做项：
1. 选 1~2 个低风险仓库或低风险任务类型
2. 指定试点负责人、review 责任人
3. 明确禁区：生产部署、生产写入、安全敏感变更默认不开放
4. 确定最小验证命令
5. 准备统一提示词模板与输出格式
6. 在仓库根目录补最小版 `AGENTS.md`

**产出物：**
- 试点范围说明
- 最小 `AGENTS.md`
- 模板入口页
- review / 验收清单

## 2.2 阶段 1：单任务执行

### 步骤 1：定义任务

**【基于已有材料的归纳整理】**
任务单至少写清：
- 目标
- 上下文
- 修改范围
- 禁止项
- 验收标准
- 工具边界
- 输出格式

### 步骤 2：先 plan，后 execute

**【直接沿用已有材料的结论】**
统一要求代理先输出：
1. 问题理解
2. 执行计划
3. 风险点

没有可接受计划，不进入执行。

### 步骤 3：限制范围

**【基于已有材料的归纳整理】**
执行时默认遵守：
- 只改相关模块
- 不做无关重构
- 不自动引入新依赖，除非明确允许
- 不提交、不推送、不发布
- 遇到高风险动作先停下说明

### 步骤 4：完成最小验证

**【直接沿用已有材料的结论】**
至少完成以下一项或多项：
- 相关测试
- lint / typecheck / build
- 复现前后对比
- 截图、日志或 diff 说明

### 步骤 5：进入 review

**【基于已有材料的归纳整理】**
review 至少检查：
- 是否符合任务范围
- 是否有无关改动
- 是否完成最小验证
- 风险是否暴露清楚
- 是否误把实践建议写成官方事实

### 步骤 6：沉淀经验

**【新增实践建议 / 待复核判断】**
每轮试点至少沉淀一种：
- 提示词模板更新项
- `AGENTS.md` 更新项
- checklist
- 可 skill 化的高频流程

## 2.3 阶段 2：团队复用

**【基于已有材料的归纳整理】**
当单任务跑通后，再逐步扩展：
1. 接入 GitHub review 做首轮筛查
2. 固化 2~3 个高频任务模板
3. 明确审批边界与责任链
4. 评估 MCP、浏览器自动化、设计实现类 skill

**【新增实践建议 / 待复核判断】**
不要把“能跑一次”直接升级成“默认团队标准”；至少经过一次复盘后再纳入 SOP。

## 3. 角色分层与职责

## 3.1 工程角色

**目标**
- 在低风险任务中稳定使用 Codex 提效
- 控制改动范围并完成验证
- 把结果交给 review，而不是绕过 review

**建议负责**
- 修小到中等 bug
- 补测试、补文档
- 做定向重构
- 处理 review comments

## 3.2 产品 / 管理角色

**目标**
- 把模糊目标转成结构化任务
- 判断哪些任务适合交给 Codex
- 写清验收标准和协作边界

**建议负责**
- 任务拆解
- 验收条件定义
- 试点范围与风险边界确认
- 培训材料与导读组织

## 3.3 评审 / 质量角色

**目标**
- 防止越权、漏审、误发布
- 保证结果可追溯、可验证、可回查

**至少检查**
- 是否先 plan 后 execute
- 是否有最小验证
- 是否有范围说明
- 是否误用事实标签
- 是否具备进入 PR / 发布下一步条件

## 3.4 推荐责任链

```text
需求提出 / 任务定义
-> 执行前计划确认
-> 执行
-> 验证
-> review
-> 合并 / 发布
-> 经验沉淀
```

## 4. 审核与风险控制

## 4.1 默认风险观

**【基于已有材料的归纳整理】**
团队默认采用：

> **默认收紧权限，按场景逐步放开，而不是默认全开。**

## 4.2 风险分级

### L1：低风险
- 读代码
- 解释代码
- 改小范围代码
- 跑本地验证

### L2：中风险
- 跨多个模块改动
- 安装依赖
- 浏览器自动化
- 只读 MCP / 内部检索

### L3：高风险
- 部署
- 生产写操作
- 安全权限改动
- 大规模代码迁移
- 外部系统写入

**【新增实践建议 / 待复核判断】**
建议：L1 可默认允许；L2 需要任务级确认；L3 必须审批 + 人工复核。

## 4.3 审核基线

**review 通过前，结果默认不算完整交付。**

review 输出至少应包含：
- 改动摘要
- 验证结果
- 风险与待确认事项
- 范围说明
- 未解决问题（如存在）

## 4.4 常见失败模式

**【基于已有材料的归纳整理】**
- 把 Codex 当成“万能开发者”
- 提示词太空，范围太大
- 没有 `AGENTS.md` 或项目说明太弱
- 过早开放高风险权限
- 把 AI review 当成人工 review 替代
- 只使用、不沉淀

## 5. 落地 Checklist

## 5.1 单任务执行 Checklist

- [ ] 目标清楚
- [ ] 修改范围清楚
- [ ] 禁止项清楚
- [ ] 已输出问题理解 / 执行计划
- [ ] 已完成最小验证
- [ ] 已说明改动摘要
- [ ] 已说明风险与待确认事项
- [ ] 已进入人工 review

## 5.2 PR / 合并前 Checklist

- [ ] Diff 与原任务范围一致
- [ ] 无明显无关改动
- [ ] 关键路径测试通过
- [ ] lint / typecheck / build 状态明确
- [ ] reviewer 已确认风险可接受
- [ ] 未解决风险已写入说明

## 5.3 团队推广 Checklist

- [ ] 已明确试点团队与范围
- [ ] 已确定导读页、指南页、模板页、SOP、来源页的入口关系
- [ ] 已确定 review / 审批责任人
- [ ] 已确定高风险操作不自动开放
- [ ] 已有最小成功案例可复用
- [ ] 已明确哪些内容是内部实践，哪些是官方事实

## 附录 A. 培训组织建议

**【新增实践建议 / 待复核判断】**
单次培训建议 60~90 分钟，按 4 段组织：
1. 统一认知：Codex 是什么、边界是什么
2. 任务定义：怎样写 prompt、怎样限定范围
3. 最小工作流演示：plan → execute → verify → review
4. 风险与协作：权限分级、发布检查、经验沉淀

培训后至少留下：
- 团队导读页
- 提示词模板页
- 最小 `AGENTS.md` 模板
- review checklist
- 发布前 checklist
- 试点任务清单

## 结语

这份 SOP 的目标不是把 Codex 讲得更“全”，而是让团队更容易：
- 正确试点
- 稳定执行
- 明确 review
- 控制风险
- 持续沉淀

如果团队已经能按这份主干流程运行，再去扩展工具、入口和自动化，成功率会更高。
