## 1. Codex 是什么

### 官方已明确

根据 OpenAI 官方发布与官方仓库说明，Codex 已不是单一“代码补全”能力，而是一组面向软件工程任务的代理式产品能力，覆盖本地终端、云端任务、部分编辑器入口以及 GitHub 工作流等场景。官方材料明确提到的方向包括：

- 编写和修改代码
- 解释代码库与协助排查问题
- 运行测试、lint、类型检查等命令
- 在隔离环境中执行任务
- 通过仓库内说明文件（如 `AGENTS.md`）改善代理对项目的理解
- 通过 skills、MCP、Web search 等方式扩展能力或上下文

### 实践建议

更实用的理解方式是：

> **Codex = 模型能力 + 代理执行能力 + 工具接入层 + 团队工作流约束**

因此，评估 Codex 不应只看“模型会不会写代码”，还要看：

- 入口选型是否合适
- 仓库是否足够可理解
- 是否有测试与验证机制
- 是否设置了清晰的权限边界
- 是否把高频任务沉淀成模板、rules 或 skills

### 来源依据

- OpenAI 官方发布：Introducing Codex / Introducing upgrades to Codex / Introducing the Codex app
- 官方仓库：`openai/codex` README

## 2. Quick Start：5 分钟建立基本认知

### 你先记住这 4 句话

1. **CLI 是本地工程师主入口。** 适合读改跑本地仓库。  
2. **Web / Cloud 更适合异步委派和并行任务。** 适合把任务“丢出去跑”。  
3. **IDE Extension 适合围绕当前文件或当前模块快速协作。**  
4. **App 更像多代理调度台。** 适合长期并行、监督和线程管理。

### 推荐最小上手动作

1. 在一个低风险仓库安装并尝试 Codex CLI
2. 在项目根目录补一份最小版 `AGENTS.md`
3. 先只做“解释代码 / 提计划 / 改小问题 / 跑最小验证”
4. 所有复杂任务统一要求：**先 plan，后 execute**
5. 暂不开放部署、生产写入、批量重构

### 推荐入门组合

**个人开发者入门组合**
- Codex CLI
- 最小版 `AGENTS.md`
- 一套标准提示词模板
- 相关测试命令 / lint / typecheck 命令

**团队试点入门组合**
- Codex CLI
- GitHub review（用于 PR 首轮筛查）
- 最小 rules / 审批策略
- 2~3 个高频任务模板或 skills

### 来源依据

- `openai/codex` README
- OpenAI 官方发布关于云端任务、GitHub review、App 的说明

## 3. 如何选择入口：CLI / Web / IDE / App

> 这一节重点收紧事实边界：只把官方已经明确说过的能力写成事实；对仍可能变化的细节，单列为待复核。

### 3.1 Codex CLI

#### 官方已明确

- `openai/codex` 是 OpenAI 官方开源仓库
- 官方 README 提供了 `npm install -g @openai/codex` 等安装方式
- CLI 作为本地终端入口，面向在当前工作目录中执行代理式软件工程任务

#### 实践建议

CLI 最适合：
- 本地修 bug
- 小到中等规模改动
- 解释代码库
- 补测试、补文档、做定向重构

建议默认工作方式：
- 在项目根目录启动
- 先要求输出理解与计划
- 明确禁止高风险动作
- 修改后强制要求最小验证

#### 待正式发布前复核

- 不同安装渠道下的最新版本差异
- 最新登录方式、平台支持与调试参数细节

#### 来源依据

- `openai/codex` README
- `openai/codex` 安装文档

### 3.2 Web / Cloud

#### 官方已明确

OpenAI 官方发布明确描述了 Codex 的云端任务形态：
- 可在 ChatGPT / Codex 页面中委派任务
- 任务可在隔离环境中执行
- 支持查看日志、测试结果、diff 等执行痕迹
- 官方叙述中强调了并行任务、异步执行和基于仓库上下文的协作方式

#### 实践建议

Web / Cloud 更适合：
- 明确范围的小任务并行处理
- 长时间执行的任务
- 需要较完整执行记录的任务
- 代码审查前的初步实现或预处理

不建议一开始就把“模糊的大需求”直接交给云端代理。

- 云端环境中对仓库、依赖、网络的最新支持细节

### 3.3 IDE Extension

#### 官方已明确

OpenAI 升级说明中明确提到 IDE 入口，强调其可在编辑器环境中协助完成软件工程任务，并与 Codex 的其他入口形成协同。

IDE 入口通常更适合：
- 围绕当前已打开文件或选中代码工作

### 3.4 App

#### 官方已明确

OpenAI 官方已发布 Codex App 相关说明，核心强调点包括：
- 面向多代理、多线程或长期任务管理
- 支持更强的任务组织与监督能力
- 与 CLI / IDE 在配置或能力上存在协同关系

#### 实践建议

App 更适合：
- 同时推进多个 issue / 任务线程
- 需要长期监督的后台工作
- 需要统一观察多个代理状态的使用者

它更像“代理调度层”，不必强行替代开发者已有编辑习惯。

#### 待正式发布前复核

- 当前下载入口
- 平台支持矩阵（如 macOS / Windows 等）
- 是否已有稳定公开的自动化、线程、worktree 等完整说明页

### 3.5 快速选择建议

| 入口 | 更适合的任务 | 不宜过度期待的点 |
|---|---|---|
| CLI | 本地读改跑、解释代码、修 bug、小重构 | 默认自动理解整个复杂系统 |
| Web / Cloud | 异步委派、并行任务、长任务、留痕 | 低延迟本地反复细修 |
| IDE | 当前文件/模块快速协作 | 大范围架构级变更自动兜底 |
| App | 多代理监督、长期任务组织 | 替代所有本地工程操作 |

## 4. GitHub review、MCP、Web search、skills 应该怎么理解

### 4.1 GitHub review

#### 官方已明确

官方升级说明明确提到 Codex 与 GitHub 工作流、代码审查能力的结合，包含 PR review 方向的能力增强。

#### 实践建议

推荐把 GitHub review 定位为：
- PR 首轮自动筛查
- 帮助发现明显问题、测试缺口、风险点
- 给人工 reviewer 提供更聚焦的切入点

**不要**把它视为人工 review 的替代品，尤其在以下场景：
- 安全问题
- 架构变更
- 业务逻辑判断
- 高风险发布改动

#### 待正式发布前复核

- `@codex review` 等具体触发方式是否仍为当前推荐写法
- 不同仓库、组织或计划的开通条件

#### 来源依据

- OpenAI 官方发布：Introducing upgrades to Codex

---

### 4.2 MCP

#### 官方已明确

OpenAI 官方升级说明明确提到 Codex 支持 MCP（Model Context Protocol）方向，用于接入外部系统与上下文。

#### 实践建议

可以把 MCP 理解为：
> 一种把外部工具、内部系统、结构化数据接给代理的标准化接口层。

典型适用方向：
- 内部文档检索
- 工单系统
- 设计系统或设计资产
- 查询型数据服务
- 企业内部 API

#### 风险提醒

MCP 的价值很大，但它也是最容易让权限边界失控的一层。建议：
- 优先只读接入
- 严格区分读写权限
- 对生产系统写操作单独审批
- 明确审计责任

#### 待正式发布前复核

- 当前 CLI / App / Cloud 对 MCP 的支持差异
- 官方文档里对配置与权限模型的最新描述

#### 来源依据

- OpenAI 官方发布：Introducing upgrades to Codex

### 4.3 Web search

#### 官方已明确

官方升级说明中已将 Web search 作为 Codex 新能力方向的一部分进行说明。

#### 实践建议

Web search 最适合：
- 查官方文档
- 查 SDK / 依赖变更

使用原则：
- 默认优先官方来源

### 4.4 skills

#### 官方已明确

官方 `openai/skills` 仓库将 Agent Skills 描述为一类供代理发现和复用的能力包。其典型结构包括：
- `SKILL.md`
- `scripts/`
- `references/`
- `assets/`（可选）

#### 实践建议

把 skill 理解为：
> 面向代理的任务 SOP + 工具封装 + 资料包。

它不是传统意义上的 IDE 插件，也不是“模型自动会做的一切”。skill 的价值在于：
- 降低高频任务的波动性
- 把团队经验标准化
- 让任务触发条件、边界与步骤更可复用

#### 待正式发布前复核

- 官方仓库当前公开的分类与样例名称
- 安装方式、自动安装范围、重启要求是否有变化

#### 来源依据

- `openai/skills` README
- `skill-creator` / `skill-installer` 等样例说明

## 5. skills：代表性样例，而非稳定完整目录

> 本节只列 **代表性样例**，不把当前仓库目录写成长期稳定、完整不变的“官方总目录”。官方仓库内容会随时间调整，发布前应再次抽样核验。

### 官方已明确

官方 `openai/skills` 仓库 README 提到有不同层级/来源的 skill 组织方式，例如系统级、精选可安装、实验性内容等。

### 实践建议

对外写作时，建议把 skill 清单表达成：
- “当前仓库中可见的代表性样例”
- “适合作为理解体系的样本”
- “并非完整目录，也不保证名称和分层长期不变”

### 代表性样例

| 样例 | 用途 | 适用边界 |
|---|---|---|
| `playwright` | 浏览器自动化、截图、页面验证 | 更适合自动化验证，不等于完整测试体系 |
| `figma-implement-design` | 基于设计上下文生成前端实现 | 需要设计上下文接入，不等于覆盖全部设计流程 |
| `gh-address-comments` | 处理 PR review comments | 依赖 GitHub 权限与仓库流程 |
| `vercel-deploy` / `render-deploy` / `cloudflare-deploy` | 部署或预览相关流程 | 默认应优先 preview，不宜直接默认生产 |
| `openai-docs` | 优先查询 OpenAI 官方开发文档 | 适合官方资料检索，不代替所有产品判断 |
| `skill-creator` | 创建或整理 skill | 适合把团队 SOP 技能化 |

### 推荐理解方式

- **插件/扩展**：给平台加入口或交互能力
- **工具**：代理可直接调用的执行能力
- **skills**：让代理在某类任务上更稳定的方法包

### 来源依据

- `openai/skills` README
- 代表性 skill 的 `SKILL.md`

## 6. AGENTS.md、rules、skills 的推荐分工

### 官方已明确

官方材料已明确提到可通过仓库说明文件（如 `AGENTS.md`）帮助 Codex 理解项目；升级说明也提到规则、权限控制与外部接入等方向。

### 实践建议

推荐分工如下：

- **AGENTS.md**：项目说明、目录导航、测试入口、规范、禁区
- **rules / 审批策略 / 沙箱配置**：权限与安全边界
- **skills**：高频任务的操作模板和专项能力

### 一个好记的分层

```text
项目怎么理解 -> AGENTS.md
什么能做、什么要审批 -> rules / sandbox / approval
某类任务怎么更稳定地做 -> skills
```

### 来源依据

- OpenAI 官方发布中对 `AGENTS.md`、安全控制、MCP、工具接入的说明
- `openai/skills` README

## 7. 推荐工作流

### 工作流 A：本地快速修 bug

适合：问题明确、复现方式清楚、改动范围可控。

1. 在项目根目录启动 CLI
2. 先要求解释相关模块与可能根因
3. 明确限制：只改相关范围，不做额外重构
4. 先输出 plan，再执行修改
5. 运行相关测试或最小验证
6. 人工 review diff

### 工作流 B：云端并行做多个小任务

适合：补测试、修 lint、文案修改、多个低耦合 issue。

1. 拆成多个小任务
2. 每个任务写清验收标准
3. 并行提交
4. 按日志、测试结果、diff 做筛选
5. 合格结果再进入 PR

### 工作流 C：GitHub review + 人工复核

1. 让 Codex 做 PR 首轮 review
2. 按“必须修 / 可讨论 / 不采纳”分级
3. 由 Codex 或本地 CLI 辅助修复
4. 最终仍由人工做合并判断

### 工作流 D：设计稿到前端实现

1. 提供设计上下文
2. 显式指定相关 skill
3. 说明框架、组件库、设计 token、验收截图
4. 先完成基础实现，再做视觉对齐
5. 用截图或浏览器自动化辅助回归

### 工作流 E：文档研究与资料沉淀

1. 优先查官方资料
2. 把事实、实践建议、待验证项分开写
3. 对关键章节保留来源依据
4. 输出可发布版结构，再进入 GitHub / Notion

## 8. 推荐 skills / 工具清单

> 以下是入门阶段值得优先考虑的 **代表性** 清单，不是必须一次性全装，也不是官方固定标准包。

### 对大多数工程团队都实用

1. **Codex CLI**
   - 作为主入口
   - 负责本地解释、修改、验证

2. **GitHub review 能力**
   - 作为 PR 首轮筛查机制
   - 减轻 reviewer 的基础检查负担

3. **`playwright` 类浏览器自动化 skill**
   - 适合 UI 验证、截图、网页流程确认

4. **`gh-address-comments` 类 PR 评论处理 skill**
   - 适合 review 闭环

5. **`openai-docs` 类官方文档检索 skill**
   - 适合 API / 产品能力调研

### 团队成熟后再考虑

6. **设计实现类 skill**
   - 如 `figma-implement-design`
   - 适合设计稿到前端的稳定流程

7. **部署类 skill**
   - 如 `vercel-deploy`、`render-deploy`
   - 建议先用于 preview 场景

8. **自定义业务 skill**
   - 把你们自己的高频 SOP 固化下来
   - 例如：修 CI、生成周报、核对接口文档、处理工单

### 选择顺序建议

- 第一步：CLI + `AGENTS.md`
- 第二步：GitHub review + 1 个最常用 skill
- 第三步：再引入浏览器自动化、设计、部署或 MCP

## 9. 团队落地建议

### 第一阶段：先跑通

- 选 1~2 个低风险仓库
- 安装 CLI
- 写最小版 `AGENTS.md`
- 只开放读、局部写、测试命令
- 暂不开放部署与生产写入

### 第二阶段：形成最小 SOP

- 沉淀标准提示模板
- 接入 GitHub review
- 固化 2~3 个高频任务的 skill 或流程模板
- 建立“先 plan、后 execute、必须验证”的习惯

### 第三阶段：团队化扩展

- 引入更明确的 rules / 审批策略
- 评估 MCP 接入内部文档、设计、工单系统
- 再考虑 App、多代理、自动化与长期任务调度

### 不建议一开始就全自动的任务

- 生产数据库写入
- 安全敏感配置变更
- 大规模删除或批量重命名
- 无测试覆盖的大型架构重构
- 需要高业务判断的复杂决策

## 结论

Codex 的真正价值，不只是“帮你写一点代码”，而是帮助团队把软件工程中的一部分重复动作变成：
- 可委派
- 可验证
- 可审查
- 可逐步标准化

要把它用好，关键不只是模型能力，而是：
- 项目是否易于理解
- 任务是否拆分得足够清楚
- 权限是否默认收紧
- 验证机制是否明确
- 经验是否通过 `AGENTS.md`、rules、skills 沉淀下来

如果这些基础打得好，Codex 更像一个可管理的工程代理体系；如果这些基础薄弱，它就容易退化成“偶尔有用、但难以规模化复用”的工具。
