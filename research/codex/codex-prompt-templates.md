# Codex 提示词模板（发布版）

> 目标：提供一套适合发布、复用与团队沉淀的 Codex 提示词模板。  
> 适用范围：工程开发、代码审查、文档研究、管理协作、非工程协同场景。  
> 使用原则：**短而准、边界清楚、可验证、便于 review。**

## 1. 先记住一个总结构

优秀的 Codex 提示词，不是越长越好，而是要尽量覆盖 6 个关键块：

1. **任务定义**：要做什么
2. **上下文**：相关仓库、模块、背景
3. **约束**：不能做什么
4. **验收标准**：怎样算完成
5. **工具边界**：能否联网、跑命令、调用外部系统
6. **输出格式**：希望它如何汇报

推荐骨架：

```text
任务
上下文
约束
验收标准
工具边界
输出格式
```

## 2. 通用主模板（最推荐）

```text
你是一个负责执行软件工程任务的 Codex 代理。

任务：
[用一句话说明目标]

上下文：
- 仓库/目录：[路径或仓库说明]
- 相关模块：[模块/文件/功能点]
- 当前问题：[bug / 需求 / 背景]
- 已知信息：[日志、复现步骤、接口约束、设计要求等]

约束：
- 只修改：[允许改动的范围]
- 不要修改：[禁止改动的范围]
- 不要引入新依赖，除非我明确允许
- 不要执行发布、提交、推送、生产写操作
- 如需做高风险操作，先停下来说明原因并等待确认

验收标准：
- [功能/问题] 被解决
- 相关测试通过 / 至少运行 [指定命令]
- lint / typecheck / build 满足 [指定要求]
- 输出根因、改动摘要、验证结果、剩余风险

工具边界：
- [允许/不允许] 联网
- [允许/不允许] 安装依赖
- [允许/不允许] 运行测试
- [允许/不允许] 使用浏览器自动化
- [允许/不允许] 调用外部系统 / MCP

输出格式：
请按以下结构输出：
1. 问题理解
2. 执行计划
3. 实际改动
4. 验证结果
5. 风险与待确认事项

要求：
- 如果任务较复杂，先只输出执行计划，不要立即修改
- 如果信息不足，先提出最小必要问题
- 如果发现更大范围问题，先报告，不要擅自扩大改动范围
```

## 3. 模板 1：先分析，不立即改代码

适合：问题复杂、需求含糊、风险较高时。

```text
请先阅读相关代码并完成分析，不要立即修改。

任务：分析 [问题/需求] 的根因与可行修复方案。

上下文：
- 相关目录：[
]
- 现象：[现象]
- 复现方式：[步骤]
- 已尝试方案：[可选]

约束：
- 本轮不要修改代码
- 不要安装依赖
- 不要做破坏性操作

验收标准：
输出应包含：
1. 根因分析
2. 涉及的文件和模块
3. 推荐修复方案（至少 1 个，最好 2 个）
4. 每个方案的影响范围与风险
5. 建议的最小改动路径
```

## 4. 模板 2：定向修 bug

```text
任务：修复 [具体 bug 名称]。

上下文：
- 问题表现：[描述]
- 复现步骤：
  1. ...
  2. ...
  3. ...
- 相关文件可能包括：[文件/目录]

约束：
- 只修改与该 bug 直接相关的代码
- 不要做无关重构
- 不引入新依赖
- 不修改公共 API，除非修复必须
- 不提交、不推送、不发布

验收标准：
- bug 能按复现步骤验证已修复
- 相关测试通过；若无测试，补最小必要测试或说明未补原因
- 输出根因、修复思路、改动文件列表、验证结果

工具边界：
- 允许读取代码和运行测试
- 如需联网或安装依赖，先说明原因

输出格式：
1. 根因
2. 修复计划
3. 改动摘要
4. 测试/验证结果
5. 风险
```

## 5. 模板 3：实现一个小功能

```text
任务：实现 [功能名称]。

背景：
- 目标用户：[谁用]
- 业务目标：[为什么做]
- 功能范围：[包含什么]
- 非目标：[不包含什么]

上下文：
- 相关模块：[目录/文件]
- 参考实现/接口：[链接或说明]
- UI/交互要求：[有则写]

约束：
- 优先复用现有组件与工具函数
- 不引入新依赖，除非确有必要并先说明
- 保持现有代码风格
- 不修改与本功能无关的模块

验收标准：
- 功能可运行
- 覆盖核心交互或核心逻辑
- 相关测试或至少最小验证步骤清楚
- 输出使用说明与后续可优化项

执行方式：
先输出实现计划和影响范围，待确认后再开始改代码。
```

## 6. 模板 4：补测试

```text
任务：为 [模块/函数/组件] 补充测试。

上下文：
- 代码位置：[路径]
- 当前缺口：[缺什么测试]
- 风险点：[边界条件/回归点]

约束：
- 尽量只新增测试文件或最小必要辅助代码
- 不重写生产代码，除非为了可测试性必须做很小调整
- 不做无关重构

验收标准：
- 覆盖核心路径与关键边界情况
- 测试可运行
- 输出覆盖了哪些场景、还缺哪些场景

输出格式：
1. 测试策略
2. 新增/修改的测试文件
3. 覆盖场景列表
4. 运行结果
5. 未覆盖风险
```

## 7. 模板 5：重构但不改变行为

```text
任务：重构 [模块/文件]，目标是提升可读性/可维护性，不改变对外行为。

上下文：
- 代码位置：[路径]
- 当前问题：[耦合高、重复代码、命名差、结构复杂等]

约束：
- 不改变外部接口行为
- 不引入新依赖
- 不扩大范围到无关模块
- 若某个改动可能影响行为，必须单独说明

验收标准：
- 逻辑行为保持一致
- 结构更清晰
- 相关测试通过
- 输出重构前后对比与风险点

要求：
先列出准备做的重构项，再开始执行。
```

## 8. 模板 6：做代码审查 / review

```text
请对以下改动做一次严格代码审查。

审查目标：
- 正确性
- 回归风险
- 安全问题
- 可维护性
- 测试缺口

上下文：
- 变更范围：[PR / diff / 文件]
- 业务目标：[这次改动是为了什么]

审查要求：
- 优先指出高影响问题
- 区分：必须修复 / 建议优化 / 可忽略
- 每个问题说明原因、影响、建议修复方向
- 如果没有明显问题，也要说明已检查了哪些方面

输出格式：
1. 总结结论
2. 必须修复的问题
3. 建议优化项
4. 测试与验证建议
5. 风险等级
```

## 9. 模板 7：查官方文档并输出研究笔记

```text
任务：研究 [主题]，优先使用官方文档与高可信来源，输出结构化中文结论。

来源要求：
- 第一优先级：官方文档 / 官方公告 / 官方 GitHub README
- 第二优先级：可信开源仓库文档
- 第三优先级：可信博客（仅作补充，必须标注）

输出要求：
- 明确区分：官方事实 / 社区经验 / 待验证项
- 给出安装或接入步骤
- 给出适用场景、优缺点、边界
- 给出来源清单

约束：
- 不要把未经核实的社区说法写成事实
- 如果信息冲突，明确列出冲突点

输出格式：
1. 结论摘要
2. 官方能力
3. 社区实践/补充
4. 安装与使用方式
5. 风险与边界
6. 待验证项
7. 来源
```

## 10. 模板 8：Figma 设计稿转前端实现

```text
任务：根据设计稿实现前端代码。

上下文：
- Figma 链接/节点：[链接]
- 技术栈：[React/Vue/Next 等]
- 组件库/设计系统：[如有]
- 目标页面/组件：[说明]

约束：
- 优先复用现有设计系统组件
- 尽量保持视觉一致
- 不随意新增 UI 库
- 不做与当前设计无关的交互扩展

验收标准：
- 布局、字体、颜色、间距尽量对齐设计稿
- 响应式行为符合要求
- 输出与设计差异说明

工具边界：
- 可使用 Figma 相关 skill / MCP
- 可使用浏览器截图进行对比验证

输出格式：
1. 实现计划
2. 设计理解
3. 代码改动摘要
4. 视觉对齐说明
5. 剩余差异与风险
```

## 11. 模板 9：自动化浏览器任务

```text
任务：使用浏览器自动化完成以下操作：[目标]

步骤目标：
1. 打开 [页面]
2. 执行 [操作]
3. 获取 [信息/截图/结果]

约束：
- 不登录未知账户
- 不提交真实生产数据
- 只执行我明确要求的页面操作
- 页面变化较大时先重新获取快照再继续

验收标准：
- 给出操作结果
- 提供截图或关键页面状态说明
- 如失败，说明失败步骤与原因
```

## 12. 模板 10：生成发布前检查清单

```text
任务：基于当前改动，生成一份发布前检查清单。

目标：
- 检查功能完整性
- 检查测试覆盖与回归风险
- 检查配置与环境变量
- 检查部署和回滚准备

输出要求：
- 按“必须完成 / 建议完成 / 可选”三级分类
- 每一项给出检查原因
- 尽量贴近当前项目结构和改动内容
```

## 13. 模板 11：管理者版（非直接写代码）

适合：研发负责人、项目经理、团队管理者，希望让 Codex 先做拆解、评估、检查，而不是直接下手改代码。

```text
你现在不是直接编码执行者，而是一个软件工程任务分析与推进助手。

任务：
请基于以下目标，输出一份适合交给工程同学或 Codex 执行代理的任务方案。

目标：
[例如：在本周内把注册流程的主要稳定性问题降下来]

上下文：
- 当前项目/产品：[说明]
- 当前问题：[说明]
- 已知限制：[时间、人力、上线窗口、依赖团队]
- 当前已有信息：[日志、用户反馈、PR、事故复盘、路线图等]

你的输出不要直接写代码，而要完成：
1. 问题拆解
2. 优先级排序
3. 建议的任务分包
4. 每个任务的验收标准
5. 风险与依赖
6. 建议由谁执行（人工 / Codex / 联合执行）

约束：
- 不要假设不存在的信息
- 不要把“可能”写成“确定”
- 对需验证的地方，单列为待确认项
- 输出应能直接拿去做排期或交付给执行代理

输出格式：
1. 目标理解
2. 优先级判断
3. 任务拆解表
4. 风险与依赖
5. 建议执行顺序
6. 待确认事项
```

### 管理者版的典型用途

- 先做需求拆解再分派
- 对一个问题做“先评估、后执行”
- 生成 PRD / 工单 / Sprint 任务草案
- 生成 review 清单、发布前检查项、回滚预案

## 14. 模板 12：非工程版（运营 / 产品 / 业务同学可用）

适合：不熟悉代码结构，但希望借助 Codex 参与资料整理、问题归纳、交付说明、验收清单等工作。

```text
请把自己当作一个能协助软件项目推进的智能助手，但本轮不要假设你可以直接替我做生产发布。

我要解决的事情：
[一句话描述目标]

背景：
- 当前在做什么：[项目/活动/版本]
- 我遇到的问题：[说明]
- 我已经有的材料：[文档、截图、会议纪要、需求、报错信息]
- 我希望最终得到什么：[汇报稿、任务清单、验收清单、问题归因、沟通提纲等]

请你帮助我：
1. 把问题说清楚
2. 把材料整理成结构化输出
3. 识别还缺什么信息
4. 形成下一步行动建议

要求：
- 尽量用非工程人员也能理解的语言
- 区分“已经确定的信息”和“还需要研发确认的信息”
- 如果涉及技术实现，不要编造细节
- 输出要适合直接发给研发、管理者或同步到文档系统

输出格式：
1. 事情概述
2. 当前已知信息
3. 需要研发/技术确认的问题
4. 建议下一步动作
5. 可直接复用的沟通文案
```

### 非工程版的典型用途

- 把需求、反馈、报错整理成研发可执行任务
- 生成发布说明、复盘提纲、会议纪要摘要
- 生成跨团队沟通文案
- 先把问题结构化，再交给工程代理执行

## 15. 高质量提示词的写法清单

### 建议写
- 明确目标
- 明确范围
- 明确不做什么
- 明确验证方式
- 明确输出格式
- 明确是否允许使用某类工具

### 不建议写
- “帮我优化一下这个项目”
- “把这个都搞定”
- “你看着办”
- “顺便把能改的都改了”
- “如果需要就自己发布”

问题在于：
- 范围不清
- 风险不可控
- 验收无标准
- 容易诱发代理擅自扩大任务

---

## 16. 一个提示词是否合格：5 个自检问题

1. 是否说清楚了目标？
2. 是否限定了改动范围？
3. 是否说明了不能做的事？
4. 是否给出了完成标准？
5. 是否要求了结果汇报结构？

如果这 5 个问题里有 2 个以上答不上来，这条提示词大概率还不够稳。

## 18. 结论

对 Codex 来说，最好的提示词不是“写得像一篇文章”，而是一份**可执行任务单**。

好的提示词会让 Codex：
- 更少跑偏
- 更少越权
- 更容易验证
- 更容易 review
- 更容易沉淀为团队模板

如果团队希望长期稳定使用 Codex，最值得优先沉淀的不是超长 prompt，而是：
- 标准提示模板
- `AGENTS.md`
- skills
- review 清单
- 发布前检查表
