# Cursor 提示词模板（Pro 账号视角草稿）

> 说明：以下模板面向 **Cursor Agent** 使用，强调“可执行、可审查、可复用”。
>
> 使用建议：
> - 常规任务优先配合 `@文件 / @符号 / @Docs`
> - 复杂任务先让 Agent 出计划，再让其执行
> - 高风险操作明确要求“先审后改 / 先方案后执行”

---

## 1. 通用任务框架模板

```text
你现在是这个仓库的开发助手。

目标：<要完成的目标>
背景：<业务背景 / 出错现象 / 需求来源>
范围：仅限 <目录 / 模块 / 文件>
约束：
- 遵守项目现有风格
- 若发现不确定点，先列假设
- 不要做无关重构
- 先给执行计划，再开始修改
验收标准：
- <标准1>
- <标准2>
- <标准3>
输出要求：
1. 先给计划
2. 再执行
3. 最后给出改动摘要、风险点、验证方式
```

适用：绝大多数 Agent 任务。

---

## 2. Bug 排查模板

```text
请帮我排查这个问题，并按“现象 -> 可能原因 -> 定位路径 -> 修复方案 -> 验证方式”输出。

问题现象：<粘贴报错 / 描述现象>
相关范围：<模块 / 接口 / 页面>
已知信息：
- <信息1>
- <信息2>
- <信息3>
约束：
- 先不要大改代码
- 先定位根因
- 如果有多个可能原因，请按概率排序
- 修复前先说明为什么这样改
```

可追加：`@报错文件 @相关组件 @测试文件`

---

## 3. 小步安全改动模板

```text
请对这个需求做最小改动实现：<需求>

要求：
- 只修改必要文件
- 不改变现有公开接口，除非必须
- 不顺手做无关清理
- 改完后列出具体改动点
- 如果需要新增测试，请补最小必要测试
```

适用：Pro 用户控制成本时很有用，避免 Agent 扩大改动面。

---

## 4. 重构模板

```text
请帮我重构 <模块/文件>，目标是：<目标>

要求：
- 先分析当前问题
- 给出重构计划与边界
- 说明哪些地方不改
- 重构后保持行为一致
- 如果存在行为变化风险，请单独标记
- 最后输出：改动摘要 / 风险 / 建议补充测试
```

建议：复杂重构先用更强模型，再切回 Auto 做收尾。

---

## 5. 测试补全模板

```text
请为 <函数/模块/接口> 补测试。

要求：
- 先阅读现有测试风格
- 沿用当前测试框架和目录结构
- 先列出需要覆盖的场景
- 至少覆盖：正常路径、边界条件、异常路径
- 不要伪造不存在的 helper
- 最后说明还有哪些场景暂未覆盖
```

## 6. 文档生成模板

```text
请基于当前代码与已有说明，为 <功能/模块> 生成一份中文文档。

输出结构：
1. 功能说明
2. 关键流程
3. 输入输出
4. 依赖与限制
5. 常见问题
6. 示例

要求：
- 只写代码中能确认的内容
- 不能确认的部分请标为“待确认”
- 尽量引用具体文件或函数名作为依据
```

## 7. 方案设计模板

```text
请先不要改代码，先给出方案。

目标：<目标>
当前约束：
- <约束1>
- <约束2>
- <约束3>

请输出：
1. 问题理解
2. 方案候选（至少 2 个）
3. 各方案优缺点
4. 推荐方案
5. 需要修改的文件范围

```

适用：高成本模型使用前，先做“规划回合”。

```text
请根据这个项目现状，帮我起草一条 Cursor Project Rule。

目标：<例如 React 组件规范 / API 校验规范 / 数据库迁移规范>
适用范围：<哪些目录 / 文件>
想约束的内容：
- <约束1>
- <约束2>
- <约束3>

输出要求：
- 直接给出可保存到 .cursor/rules/ 的 markdown 内容
- 若适合，请补充 frontmatter（description / globs / alwaysApply）

```

## 10. MCP 使用模板

```text
请优先使用已配置的 MCP 工具完成这项任务：<任务>

要求：
- 先说明你准备调用哪些 MCP 工具
- 如果涉及写操作，先征求确认
- 调用后总结：用了哪些工具、拿到了什么信息、还缺什么
- 如果 MCP 不足以完成，再退回本地搜索或手工实现
```

适用：避免 Agent 乱调用外部系统。

## 11. Hooks / 风险控制模板

```text
我想为 Cursor 增加一个 hook，用于 <目标，例如阻止危险 shell 命令 / 编辑后自动格式化 / 记录审计日志>。

```

## 12. 成本控制模板

```text
请按“成本优先”的方式完成这项任务：<任务>

要求：
- 优先使用较省额度的方式获取信息
- 先搜索并缩小范围，再做深度分析
- 尽量减少无关文件读取
- 只有在必要时才建议切换到更强模型
- 最后说明：如果要进一步提高正确率，下一步建议切到什么模型
```

适用：Pro 用户日常非常实用。

## 13. 审查与收尾模板

```text
请不要继续开发，只做审查与收尾。

请输出：
1. 改动摘要
2. 受影响文件列表
3. 潜在风险
4. 是否需要补测试
5. 是否有超出原需求的改动
6. 建议我如何人工复核
```

---

## 14. 一组推荐组合打法

### 14.1 低成本日常开发
- 模型：Auto
- 模板：通用任务框架 + 小步安全改动

### 14.2 复杂问题处理
- 先用：方案设计模板
- 再用：Bug 排查 / 重构模板
- 模型：更强模型优先

### 14.3 规范沉淀
- 用 Rules 草稿模板产出 `.cursor/rules/`
- 用 Skill 草稿模板沉淀高频工作流

### 14.4 集成自动化
- 先用 MCP 使用模板
- 再用 Hooks / 风险控制模板

---

## 15. 发布前提醒
- 模板里的“先给计划再执行”适合大多数复杂任务
- 若任务是非常小的改动，可以去掉计划阶段，减少对话轮次
- 若项目已有 rules / skills，提示词要避免与已有约束冲突
