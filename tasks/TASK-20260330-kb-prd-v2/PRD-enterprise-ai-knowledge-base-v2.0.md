- 文档版本：v2.0
- 文档状态：已正式发布
- 适用阶段：产品需求定义 / 方案落地前评审
- 输出日期：2026-03-30
- 适用读者：产品、研发、架构、信息中心、信息安全、管理层、运维

## 1. 文档概述

### 1.1 项目背景
当前企业内部 AI 知识问答系统已经完成基础能力打通，员工主要通过飞书发起问答，系统能够基于内部文档内容返回答案与引用来源。

但在当前链路下，用户虽然能够看到回答与部分引用信息，仍无法在飞书中稳定、受控地打开对应的**真实原始 PDF**。这导致以下核心问题长期存在：

1. 用户无法快速核对原文，答案可信度不足。
2. 引用来源与真实原始文件之间缺少稳定补链，无法形成完整追溯链路。
3. 即使 FastGPT 给出引用结果，也不代表用户可受控地访问原始材料。
4. 若直接暴露文件目录、服务器路径或将原始 PDF 上传到飞书云，会带来明显安全与合规风险。
5. 当前系统已具备桥接公网与内网的能力，但“问答引用 → 原始文件访问”这一段尚未正式产品化。

因此，本次 PRD 2.0 的目标不是重建知识库系统，而是在**现有真实架构基础上，补齐“引用可信追溯 + 原始 PDF 受控打开”这一关键链路**，形成一版低风险、可上线、可审计、可演进的系统优化方案。

### 1.2 本期目标
本期目标聚焦于以下事项：

1. 在飞书回答中，为引用来源补充可点击、真实可用、受控的原始 PDF 打开能力。
2. 在不改变当前核心架构的前提下，建立引用标识与原始 PDF 的稳定映射关系。
3. 明确桥接系统在问答后处理、链接生成、访问控制、跨子公司访问中的职责。
4. 明确单文件受控访问原则，避免目录浏览、真实路径暴露和文件托管外泄。
5. 明确权限一致性、审计追溯、运维维护、后续演进路线。

### 1.3 产品定位
本次 PRD 2.0 定位为：

> 对现有企业内部 AI 知识问答系统进行一次围绕“可信引用 + 原始 PDF 受控访问”的系统补链优化。

它**不是**：

- 单纯修改 FastGPT 的前端展示
- 大规模重构知识库架构
- 替换现有 MinerU / FastGPT / vLLM / 飞书入口体系
- 将原始 PDF 发布到飞书云或公开文件系统
- 一次性解决页码级、段落级精准定位等增强能力

### 1.4 适用范围
本 PRD 适用于：

- 飞书作为唯一用户入口的内部 AI 问答场景
- 基于 PDF 原始文档转 Markdown 入库的知识问答场景
- 员工需核对问答依据、查看原始材料的场景
- 跨子公司或跨网络访问内部知识问答与原文材料的场景
- 需满足权限一致性、访问受控、审计可追溯的企业内使用场景

### 1.5 非本期范围
本期明确不纳入以下内容：

1. 开放整库目录浏览或文件树浏览。
2. 向用户暴露服务器真实文件路径、共享盘路径、NAS 路径。
3. 将原始 PDF 存入飞书云文档或飞书云盘作为正式方案。
4. 依赖大模型自由生成真实文件链接。
5. 重构 FastGPT、MinerU、vLLM 的底层架构或部署关系。
6. 推动 MinerU 在本期迁移到内网服务器。
7. 页码级、段落级、章节级精确定位。
8. 支持原始 Office、图片、视频等更多文件类型的原文打开。
9. 建设面向公网的开放下载系统。

### 1.6 术语定义
- **原始 PDF**：业务上真实存档、供核对的 PDF 原文件。
- **Markdown 入库文件**：由 MinerU 从 PDF 转换后，实际进入知识库平台的 Markdown 内容。
- **引用标识**：FastGPT 在检索/回答结果中返回、可用于定位引用来源的字段，如 collection_id、source_name、sourceId 等；具体稳定字段待验证。
- **映射关系**：Markdown 文档、FastGPT 记录、原始 PDF 之间的关联数据。
- **桥接系统**：部署于 Windows 工作电脑的现有中间层，负责连接飞书公网与局域网内服务，并承担跨子公司访问能力。
- **受控访问链接**：由系统生成、仅用于打开单个 PDF、带鉴权/签名/时效控制的访问地址。
- **权限基线**：判断某员工是否应当有权打开某份 PDF 的权威来源规则。

## 2. 已确认事实、方案假设、待验证项

### 2.1 已确认事实
以下内容已由当前任务上下文确认，可作为本 PRD 的事实基础：

1. 当前文档处理链路为：**MinerU 将 PDF 转为 Markdown，Markdown 作为唯一入库格式**。
2. 当前知识库平台为 **FastGPT**，负责切片、向量化、检索、问答生成。
3. 当前模型层为 **vLLM**，并由桥接系统控制休眠/唤醒。
4. **FastGPT、vLLM、内网知识服务部署在本地内网服务器**。
5. **桥接系统部署于 Windows 工作电脑**，负责连接飞书公网与内网服务，并支撑跨子公司访问。
6. **飞书是唯一用户入口**，用户不直接进入 FastGPT 页面。
7. 本次核心目标是：**在飞书中提供可靠、受控、可打开整份原始 PDF 的链接**。
8. 本期不允许：开放目录浏览、暴露真实文件路径、将原始 PDF 托管到飞书云、依赖模型生成真实链接。
9. 本期不引入大规模架构重构，优先低风险、简单、快速上线。

### 2.2 方案假设
以下内容是为形成可落地方案而提出的合理假设，需在立项/开发前进一步确认：

1. 桥接系统可新增或挂接一层轻量的“引用后处理 + 单文件访问控制”能力，而无需重构现有问答主链路。
2. 文档入库流程中可以补充一个“映射记录写入”动作，用于维护 Markdown / FastGPT / PDF 之间的关联。
3. 飞书入口可带出可识别用户身份信息，供桥接系统做鉴权或权限校验。
4. 内网中存在或可新增一个受控文件访问能力，由桥接系统统一代理对外，不直接暴露文件系统。
5. 原始 PDF 存储位置在业务上相对稳定，至少可被映射系统可靠引用，不要求本期彻底统一底层存储介质。

### 2.3 待验证项 / 待确认项
以下内容不得在项目启动前被误写为已确认事实：

1. **FastGPT 稳定可返回哪些引用字段**，以及这些字段在不同问答模式下是否一致，属于待验证项。
2. **PDF 打开权限最终以谁为准**，需在业务权限、知识库权限、飞书身份、存储侧权限之间确定权威基线。
3. 桥接系统当前是否已具备签名链接、短时令牌、流式代理下载等能力，需技术确认。
4. 跨子公司访问链路中，飞书身份能否直接映射到内部权限主体，需确认。
5. 原始 PDF 的实际存储形态、统一命名规范、是否存在重复版本、多位置副本，需梳理。
6. 如果引用结果只返回文件名而无稳定 ID，映射碰撞风险如何消解，需验证。
7. 访问原始 PDF 时是“在线预览”还是“下载后二次打开”为主，需结合终端体验与安全策略确认。
8. 审计日志最终落在哪个系统、留存周期如何配置，需与运维/安全侧确认。

## 3. 当前真实系统架构与运行链路

### 3.1 当前真实架构概览
基于已确认事实，当前系统的真实架构如下：

1. **文档处理层**：MinerU 将原始 PDF 转换为 Markdown。
2. **知识入库层**：Markdown 作为唯一入库格式进入 FastGPT。
3. **知识问答层**：FastGPT 完成切片、向量化、检索、问答生成。
4. **模型服务层**：vLLM 提供模型推理能力，并由桥接系统控制休眠/唤醒。
5. **内网服务层**：FastGPT、vLLM、内网知识服务部署在本地内网服务器。
6. **桥接层**：Windows 工作电脑上的桥接系统负责连接飞书公网与内网服务，并承接跨子公司访问。
7. **用户入口层**：飞书是唯一用户访问入口，用户不直接打开 FastGPT 页面。

### 3.2 当前运行链路
当前从问答到返回的主链路可描述为：

1. 用户在飞书内发起问题。
2. 飞书请求进入桥接系统。
3. 桥接系统按现有逻辑访问内网服务，并唤醒或调用 vLLM / FastGPT。
4. FastGPT 基于已入库的 Markdown 内容完成检索与答案生成。
5. FastGPT 返回答案及部分引用相关结果。
6. 桥接系统将整理后的问答结果返回飞书。

### 3.3 当前缺口所在位置
当前缺口不是“问答能力没有”，而是链路中缺少以下一段：

> **引用结果 → 真实原始 PDF → 单文件受控打开**

更具体地说，当前链路虽然已经能返回问答和一定程度的引用，但没有形成以下闭环：

1. 没有稳定、系统化的引用标识到 PDF 的映射机制。
2. 没有面向飞书用户的单文件受控访问入口。
3. 没有把权限一致性与访问审计纳入这一补链能力。
4. 没有把跨子公司访问下的原文打开能力纳入统一桥接出口。

### 3.4 为什么这是“系统补链优化”而不只是改 FastGPT
本问题不能被定义为“给 FastGPT 加一个引用链接”，原因如下：

1. **原始 PDF 不在 FastGPT 的原生展示闭环内**。FastGPT 处理的是已入库内容，不等于它天然管理原始文件受控访问。
2. **飞书才是唯一用户入口**。最终用户体验发生在飞书侧，必须由桥接层完成对飞书返回结果的产品化补链。
3. **权限控制不应交给模型自由生成**。真实 PDF 链接必须由系统生成、鉴权、审计，而非依赖模型拼接文本。
4. **跨子公司访问由桥接系统承担**。若不经过桥接系统，原始 PDF 无法自然复用现有公网到内网的连接能力。
5. **安全边界在 FastGPT 之外**。单文件受控访问、路径隐藏、审计留痕，本质上属于系统访问控制能力，而非单一问答引擎能力。
6. **映射维护发生在入库与运维流程中**。这需要文档处理、入库、引用返回、访问控制四段协同，不是单点前端改造。

结论：本次需求的本质是现有 AI 知识问答系统的一次**补链型系统优化**，FastGPT 只是其中一个组件，而不是全部答案。

## 4. 业务问题与价值定义

### 4.1 当前核心问题
当前系统面临的核心问题为：

1. 飞书中虽然能给出回答，但用户对答案依据的核对成本高。
2. 引用结果无法稳定回到真实原始 PDF，造成“可回答、难验证”。
3. 缺少受控访问机制时，系统要么不给原文，要么只能通过高风险方式暴露原文。
4. 权限边界未被统一表达：能问到、能看到引用，不代表一定能打开原 PDF。
5. 跨子公司场景下，若原文访问链路独立建设，会造成新的网络和权限割裂。

### 4.2 本期业务价值
本期优化完成后，将带来以下业务价值：

1. **可信性提升**：用户可直接核对原始 PDF，降低对 AI 回答的盲信风险。
2. **追溯性补齐**：从飞书答案可追到真实原文材料，形成完整业务链路。
3. **体验闭环形成**：问、答、引、查不再断裂。
4. **安全性提升**：在不开放目录、不暴露路径、不上传飞书云的前提下，实现受控访问。
5. **跨组织可用性提升**：跨子公司场景下，原 PDF 打开能力纳入统一桥接链路，减少新的孤立访问入口。
6. **后续扩展基础形成**：为页码定位、更多文件类型、自动化映射打下基础。

### 4.3 本期成功标准
本期成功不以炫技为目标，而以“低风险补链上线”为目标。成功标准包括：

1. 飞书回答中可展示来源，并支持打开对应原始 PDF。
2. 打开行为仅限单文件，不支持目录级浏览。
3. 链接由系统生成，不依赖模型自由生成。
4. 不暴露真实服务器路径或存储路径。
5. 跨子公司访问场景下，授权用户可通过桥接路径完成访问。
6. 未授权用户无法通过链接绕过权限直接打开文件。
7. 有最小可用审计日志，能追踪谁在何时打开了哪份文件、结果如何。
8. 映射关系具备维护方式，新增文档不会使系统失控。

## 5. 用户角色与典型使用场景

### 5.1 角色定义

| 角色 | 定义 | 本期关注点 |
| --- | --- | --- |
| 普通员工 | 在飞书中发起知识问答的内部用户 | 看答案、看来源、打开原 PDF 核对 |
| 部门负责人 / 专业人员 | 对答案准确性要求高的业务角色 | 追溯原文、确认制度与规范 |
| 文档维护人 | 负责 PDF、Markdown、入库流程的人 | 维护映射、处理版本与更新 |
| 系统管理员 | 负责桥接系统、内网服务与稳定性 | 配置访问控制、监控、故障处理 |
| 安全/审计角色 | 负责核查越权与访问留痕 | 查看审计日志、追踪访问链路 |

### 5.2 典型场景

#### 场景 A：员工核对 AI 回答依据
1. 员工在飞书提问。
2. 系统返回答案并展示引用来源。
3. 员工点击“打开原 PDF”。
4. 系统校验权限后打开对应文件。
5. 员工核对原文并确认答案是否可信。

#### 场景 B：跨子公司员工访问共享制度文件
1. 员工位于非总部网络或跨子公司网络环境。
2. 员工通过飞书提问并获得引用来源。
3. 桥接系统沿现有跨网络能力返回受控文件访问链接。
4. 用户在授权前提下打开文件，无需额外记忆内网地址或文件路径。

#### 场景 C：安全核查一次争议回答
1. 某回答被质疑依据不充分。
2. 管理者或安全角色根据引用结果与审计记录回溯。
3. 可确认系统当时引用了哪份 PDF、由谁发起访问、访问结果如何。

## 6. 产品目标与设计原则

### 6.1 产品目标
1. 为飞书回答补齐原始 PDF 受控打开能力。
2. 建立引用结果到原始 PDF 的稳定映射关系。
3. 实现权限一致、路径隐藏、访问可审计。
4. 维持现有系统架构稳定，不引入重构级风险。

### 6.2 设计原则

#### 原则 1：基于真实现状，不重写现有架构
本期仅在现有 MinerU、FastGPT、vLLM、桥接系统、飞书入口之上补链，不以“重新设计全系统”为前提。

#### 原则 2：单文件受控访问，不开放目录
用户只能访问系统明确返回的那一个 PDF，不应看到目录、相邻文件或底层文件树。

#### 原则 3：链接必须由系统生成，不由模型生成
模型只能返回引用信息，不能被赋予生成真实文件地址的职责。

#### 原则 4：权限校验必须独立于模型回答
即使模型引用到了某文件，系统仍需对“打开原 PDF”动作单独做鉴权。

#### 原则 5：优先复用桥接系统
桥接系统已是飞书到内网的事实中枢，本期优先以其作为后处理和访问控制统一出口。

#### 原则 6：显式区分已确认事实与待验证项
对于 FastGPT 引用字段、权限基线来源等不确定内容，必须在方案中保留待验证口径。

## 8. 总体方案设计

### 8.1 总体方案概述
本期建议方案为：

1. 在文档入库阶段，建立“原始 PDF ↔ Markdown 入库文档 ↔ FastGPT 引用标识”的映射关系。
2. 在飞书问答返回前，由桥接系统对 FastGPT 引用结果进行后处理。
3. 桥接系统基于映射关系，为每个可识别引用来源生成“受控 PDF 打开链接”或等效访问令牌。
4. 用户点击链接后，请求仍通过桥接系统或桥接系统可控的受限文件访问服务。
5. 系统对该次打开请求进行身份识别、权限校验、审计记录。
6. 校验通过后，仅返回对应单个 PDF 的预览/下载流，不提供目录能力。

### 8.2 目标架构文字图

```text
员工（飞书）
   ↓
飞书消息入口
   ↓
桥接系统
   ├─ 调用 FastGPT 获取答案与引用结果
   ├─ 调用映射关系服务/映射表
   ├─ 对引用结果做后处理并生成受控 PDF 链接
   └─ 统一承接跨子公司访问与鉴权
   ↓
飞书返回：答案 + 引用来源 + 打开原 PDF
   ↓（用户点击）
桥接系统 / 受限文件访问服务
   ├─ 身份识别
   ├─ 权限校验
   ├─ 审计记录
   └─ 单文件流式返回 PDF
   ↓
原始 PDF 存储位置（内网）
```

### 8.3 方案关键点
1. 引用补链在桥接层完成，而非要求模型生成真实地址。
2. 文件访问通过受控服务完成，而非对文件目录做直接映射。
3. 访问路径对用户可见的是业务链接，不是物理存储路径。
4. 权限校验发生在“打开文件”这一动作上，而不是默认继承问答结果。
5. 统一复用桥接系统承担跨子公司访问能力。

## 9. 引用标识与原始 PDF 映射关系设计

### 9.1 设计目标
映射机制要解决的问题是：

> 当 FastGPT 返回一条引用来源后，系统如何稳定找到唯一对应的原始 PDF，并保证该映射可维护、可审计、可更新。

### 9.2 映射对象
本期建议至少维护以下三类对象的关联：

1. **原始 PDF 文件对象**
2. **Markdown 入库文档对象**
3. **FastGPT 可返回的引用对象**

### 9.3 推荐映射模型
建议建立一张或一组映射记录，核心关系如下：

| 字段 | 说明 | 状态 |
| --- | --- | --- |
| mapping_id | 映射记录唯一 ID | 推荐新增 |
| source_pdf_id | 原始 PDF 内部唯一标识 | 推荐新增 |
| source_pdf_name | 原始 PDF 展示名 | 推荐新增 |
| source_pdf_version | PDF 版本号/版本标识 | 推荐新增 |
| source_pdf_storage_ref | PDF 存储引用，不暴露真实路径 | 推荐新增 |
| markdown_doc_id | Markdown 入库文档 ID | 推荐新增 |
| markdown_doc_name | Markdown 文档名 | 推荐新增 |
| fastgpt_dataset_id | FastGPT 知识库/数据集标识 | 待结合实际确认 |
| fastgpt_collection_id | FastGPT 返回的集合/文件标识 | 待验证 |
| fastgpt_source_id | FastGPT 返回的来源标识 | 待验证 |
| fastgpt_source_name | FastGPT 返回的来源名称 | 待验证 |
| business_scope | 业务归属 / 公司 / 部门范围 | 推荐新增 |
| permission_baseline_ref | 权限基线引用 | 推荐新增 |
| status | 生效、停用、待修复 | 推荐新增 |
| created_at / updated_at | 建档与更新时间 | 推荐新增 |
| operator | 最后维护人 | 推荐新增 |

### 9.4 映射唯一性原则
1. 不以“文件名文本”作为唯一可信主键。
2. 若现阶段无法直接获得 FastGPT 稳定 ID，则需采用“文件名 + 数据集 + 版本 + 人工确认”的过渡策略。
3. 同名文件、多版本文件必须能区分，避免引用误指。
4. 原始 PDF 即使移动物理位置，也应通过 storage_ref 间接定位，而非将真实路径写入飞书链接。

### 9.5 映射建立时机
建议在文档入库流程中补充以下动作：

1. 原始 PDF 确定入库。
2. MinerU 转出 Markdown。
3. Markdown 入 FastGPT。
4. 获取或回填 FastGPT 可用引用标识。
5. 写入映射表。
6. 标记映射状态为“可用”。

### 9.6 映射维护原则
1. 文档新增时，映射为必填动作，不允许长期裸奔入库。
2. 文档更新时，需确认是否沿用旧 PDF、产生新 PDF 版本，或需要新映射。
3. 文档下线时，应同步停用映射，避免打开失效文件。
4. 若引用字段策略变更，应支持批量补录或修复映射。

### 9.7 待验证项说明
**FastGPT 稳定可返回的引用字段**当前属于待验证项，正式开发前需完成样本验证。PRD 中不将 collection_id、sourceId、source_name 等字段写死为既成事实，而将其定义为“候选稳定标识”。

## 10. 桥接系统后处理职责

### 10.1 为什么由桥接系统承担后处理
桥接系统已经是飞书到内网服务的既有中间层，且承担跨子公司访问职责，因此最适合作为补链能力的执行位置。

### 10.2 桥接系统新增职责
本期建议桥接系统承担以下新增职责：

1. 接收 FastGPT 问答结果与引用结果。
2. 提取可用引用标识。
3. 查询映射关系，定位原始 PDF。
4. 为飞书返回结果补充“打开原 PDF”按钮/链接。
5. 生成带时效与校验能力的受控访问链接或令牌。
6. 承接用户点击后的文件访问请求。
7. 做身份识别、权限校验、审计留痕。
8. 将 PDF 以单文件方式流式返回给用户。

### 10.3 桥接系统不应承担的职责
1. 不负责由模型自由生成真实文件路径。
2. 不直接暴露存储目录或共享盘给用户。
3. 不在本期承担完整文档管理平台的重建。
4. 不绕开权限体系直接放开所有问答命中文档访问。

### 10.4 后处理输出规则
桥接系统对问答结果的后处理，应遵循：

1. 若某条引用成功映射到原 PDF，展示“打开原 PDF”。
2. 若映射缺失，则保留来源展示，但提示“原文链接待补齐”或等效文案。
3. 若用户后续点击时权限不足，则返回明确、可理解但不泄露细节的提示。
4. 若链接已过期，则提示重新从飞书问答结果进入，不允许长期裸链传播。

## 11. 受控 PDF 访问机制设计

### 11.1 设计目标
在不开放目录、不暴露路径、不上传飞书云的前提下，让授权用户能从飞书中打开单个 PDF。

### 11.2 推荐方案
推荐采用：

> **桥接系统统一入口 + 短时效受控链接/令牌 + 服务端二次鉴权 + 单文件流式返回**

该方案的基本过程为：

1. 飞书回答返回的并不是物理文件地址，而是桥接系统签发的受控访问地址。
2. 用户点击后，请求到达桥接系统或桥接系统可控的受限文件访问服务。
3. 服务端基于用户身份、文件标识、时效、权限基线做校验。
4. 校验通过后，服务端读取目标 PDF 并以预览/下载流方式返回。
5. 全程不暴露真实服务器路径，也不返回目录信息。

### 11.3 单文件访问约束
系统必须满足以下约束：

1. 链接只能定位到单个 file_id / pdf_id。
2. 链接不能列目录、不能索引相邻文件。
3. 链接不可通过改参数枚举出其他文件。
4. 链接过期后不可继续访问。
5. 未通过校验时不得返回文件是否存在、路径是否真实等内部信息。

### 11.4 链接形态建议
建议使用如下抽象形态：

`https://bridge-domain/file/open?token=xxx`

或

`https://bridge-domain/file/{opaque_id}?sig=xxx&exp=xxx`

要求：

1. opaque_id 为不可反推出真实路径的业务标识。
2. sig 为服务端签名结果，防篡改。
3. exp 为短时效参数。
4. 访问仍需结合服务端二次鉴权，不以“拿到链接”视为必然授权。

### 11.5 打开方式建议
本期建议优先支持：

1. 浏览器内预览（若终端兼容）。
2. 受控下载打开（作为兼容兜底方案）。

具体以终端体验验证结果为准，但无论采用何种方式，访问控制与审计逻辑应保持一致。

## 12. 权限一致性与权限基线来源

### 12.1 本期必须回答的问题
本期必须明确：

> “用户是否有权打开某个 PDF，到底以谁为准？”

### 12.2 推荐方案：以业务文档权限基线为准，桥接系统做统一执行
本 PRD 推荐以下口径：

1. **权限基线应以原始文档对应的业务权限归属为准**，而不以“模型检索到了”作为授权依据。
2. 桥接系统在打开 PDF 时，统一读取或判断该文档的权限基线。
3. 飞书身份仅作为用户身份载体，不直接等于文档授权本身。
4. FastGPT 检索命中仅说明“内容被引用”，不天然授予原始文件打开权。

换言之：

- **问答命中 ≠ 原文可打开**
- **原文是否可打开 = 用户身份 + 文档权限基线 + 桥接系统校验结果**

### 12.3 推荐的权限判断顺序
建议权限判断顺序如下：

1. 识别当前访问用户的企业身份（来自飞书会话或桥接认证上下文）。
2. 识别目标 PDF 所属的业务权限范围（公司 / 子公司 / 部门 / 知识范围 / 文件分级）。
3. 基于权限基线判断用户是否在允许范围内。
4. 若存在更严格的文件级限制，以更严格规则优先。
5. 校验通过后方可打开文件。

### 12.4 权限基线来源建议
推荐优先级如下：

1. **一级推荐**：以文档归属业务系统或知识管理规则中的授权范围为准。
2. **二级兜底**：若当前无成熟业务权限系统，则在映射表中维护最小可用的访问范围字段，如 company_scope / department_scope / sensitivity_level。
3. **不推荐**：以 FastGPT 是否可返回引用作为 PDF 权限依据。
4. **不推荐**：仅凭持有链接即允许访问。

### 12.5 待确认项
以下仍需业务和技术共同确认：

1. 当前是否已有正式的文档权限源系统可对接。
2. 飞书身份与内部组织身份的映射是否已打通。
3. 若飞书问答权限与原始 PDF 权限口径不一致，最终谁优先。
4. 对跨子公司共享文件，权限判断是按集团统一身份还是按子公司独立身份。

### 12.6 本期最低落地要求
若短期内无法完成完整统一权限系统对接，本期最低要求应为：

1. 每份可打开 PDF 必须有明确的权限范围配置。
2. 桥接系统在每次打开动作上执行校验。
3. 权限不足必须拒绝访问并留痕。
4. 后续支持接入更权威的权限源时，不推翻现有受控访问框架。

## 13. 审计与可追溯设计

### 13.1 审计目标
本期审计设计的目标不是做庞大日志平台，而是保证以下问题可被回答：

1. 谁在什么时间打开了哪份原始 PDF。
2. 该访问是从哪次飞书问答上下文触发的。
3. 访问是否通过权限校验。
4. 若失败，失败原因大类是什么。
5. 一次争议回答是否能回溯到对应原文与访问记录。

### 13.2 审计日志最小字段集
本期必须至少记录以下字段：

| 字段 | 说明 |
| --- | --- |
| event_id | 事件唯一 ID |
| event_time | 事件时间 |
| user_id | 访问用户 ID |
| user_name | 用户显示名（可选，建议保留） |
| org_id / company_id | 组织或子公司标识 |
| session_id | 会话或请求链路 ID |
| source_channel | 来源渠道，固定为飞书或桥接入口 |
| qa_request_id | 问答请求 ID / 对话关联 ID |
| citation_ref | 引用标识原值 |
| mapping_id | 命中的映射记录 ID |
| pdf_id | 目标 PDF ID |
| action | 动作类型，如 open_pdf / preview_pdf / download_pdf |
| auth_result | 鉴权结果：allow / deny |
| deny_reason_code | 拒绝原因码（权限不足、链接过期、映射缺失等） |
| client_ip | 客户端出口 IP（如可获取） |
| device_info | 终端信息（如可获取） |
| trace_id | 全链路追踪 ID |

### 13.3 审计要求
1. 链接生成日志与文件打开日志建议可关联。
2. 权限拒绝同样必须记录，不能只记成功访问。
3. 审计字段不应反向暴露真实文件路径。
4. 日志查询权限应受控，仅管理员/安全角色可查。
5. 留存周期与脱敏规则需与安全/运维侧确认。

## 14. 飞书返回展示规则

### 14.1 返回结构建议
飞书中返回结果建议包含以下层次：

1. AI 回答正文。
2. 引用来源列表。
3. 每条来源对应的“打开原 PDF”操作入口（若映射成功）。
4. 无映射或无权限场景下的明确信息提示。

### 14.2 展示文案建议
- 正常：`来源：员工手册（可打开原 PDF）`
- 映射缺失：`来源：员工手册（原文链接待补齐）`
- 权限不足：`当前无法打开该原文`
- 链接失效：`访问链接已失效，请返回飞书重新打开`

### 14.3 展示边界
1. 不展示真实存储地址。
2. 不展示目录结构。
3. 不展示内部文件系统错误栈。
4. 不因提升可点击性而泄露内部实现字段。

## 15. 关键流程设计

### 15.1 入库建链流程
1. 业务侧提供原始 PDF。
2. MinerU 将 PDF 转为 Markdown。
3. Markdown 入库 FastGPT。
4. 系统获取或回填 FastGPT 可用引用标识。
5. 写入映射表：PDF ↔ Markdown ↔ 引用标识。
6. 配置该文档的权限基线范围。
7. 映射生效后，文件具备被飞书问答补链的条件。

### 15.2 问答返回补链流程
1. 用户在飞书发问。
2. 桥接系统调用 FastGPT 获取答案和引用。
3. 桥接系统提取候选引用标识。
4. 根据映射表查找原始 PDF。
5. 为命中的引用生成受控访问入口。
6. 将答案、来源、打开原 PDF 一并返回飞书。

### 15.3 原 PDF 打开访问流程
1. 用户点击飞书中的“打开原 PDF”。
2. 请求到达桥接系统/受控访问服务。
3. 校验链接签名、时效、请求完整性。
4. 识别用户身份。
5. 根据权限基线校验是否允许访问。
6. 写入审计日志。
7. 若通过，则返回单个 PDF 预览/下载流。
8. 若拒绝，则返回统一错误提示。

### 15.4 权限校验失败流程
1. 用户点击链接。
2. 系统识别权限不足、链接过期或映射失效。
3. 不返回底层文件信息。
4. 写入拒绝审计日志。
5. 返回统一文案提示，并引导用户联系管理员或重新发起问答。

## 16. 跨子公司访问链路约束

### 16.1 问题定义
当前桥接系统已承担跨子公司访问职责，因此原始 PDF 打开能力不能脱离这一现实单独建设，否则会造成新的网络断点与权限割裂。

### 16.2 本期设计约束
跨子公司访问必须遵循以下约束：

1. **统一走桥接出口**：原始 PDF 访问不应要求用户额外进入内网地址或文件共享路径。
2. **身份必须可识别**：跨子公司用户身份必须能在桥接层被识别成可校验主体。
3. **权限必须按文档范围校验**：跨子公司不等于默认可共享，仍需按文档归属范围校验。
4. **链路必须最小暴露**：无论用户来自总部还是子公司，都不应看到真实内网文件位置。
5. **失败提示应统一**：不能因为网络拓扑差异给出暴露内部结构的错误提示。

### 16.3 推荐访问路径
推荐路径为：

飞书用户 → 桥接系统公网入口 → 桥接系统鉴权/转发 → 内网受控文件访问能力 → 原始 PDF

### 16.4 待确认项
1. 子公司用户的组织身份标识是否与总部统一。
2. 子公司网络访问桥接入口的稳定性与带宽是否满足 PDF 打开需求。
3. 对涉敏文件是否允许跨子公司打开，或需额外二次审批。
4. 是否需要对子公司访问应用更短时效链接或更严格审计策略。

## 17. 运维、维护与数据治理要求

### 17.1 映射维护要求
1. 新增入库文档必须同步登记映射。
2. 映射异常应有“待修复”状态，避免静默失败。
3. 维护人应可按 mapping_id / pdf_name / fastgpt_source 查询问题。
4. 映射变更要留痕，包括变更人、时间、变更原因。

### 17.2 运维要求
1. 桥接系统需有基础可观测性：访问量、成功率、拒绝率、异常率。
2. 文件访问失败需能区分映射问题、鉴权问题、存储读取问题、链接过期问题。
3. 当内网文件位置变更时，应支持修复 storage_ref，而不影响对外链接规则。

### 17.3 数据治理要求
1. 原始 PDF、Markdown、FastGPT 引用记录之间应尽量保持版本一致。
2. 废弃文档应停止对外访问。
3. 多版本文档应有明确“当前有效版本”策略。
4. 对高敏文件建议增加分级标识，供权限基线与审计策略使用。

---

## 18. 非功能要求

### 18.1 安全要求
1. 不暴露真实文件路径。
2. 不允许目录浏览。
3. 链接需具备防篡改与短时效能力。
4. 文件打开需二次鉴权。
5. 审计可追溯。

### 18.2 稳定性要求
1. 问答补链不应显著拖慢飞书回答主体验。
2. 文件打开服务异常时，不影响问答主链路返回答案。
3. 映射服务异常时，系统至少可降级为“显示来源但不提供 PDF 打开”。

### 18.3 可维护性要求
1. 映射表结构应支持后续扩展字段。
2. 新文档接入流程应标准化，减少纯人工记忆。
3. 变更底层存储位置时不应要求重做所有用户侧链接。

### 18.4 可扩展性要求
本期设计应为后续能力预留空间：

1. 页码级定位
2. 段落级定位
3. 多文件类型支持
4. 自动化映射生成
5. 更权威权限源接入

---

## 19. 风险、假设与待确认项汇总

### 19.1 关键风险
1. FastGPT 引用字段不稳定，导致映射策略需要调整。
2. 原始 PDF 权限源不明确，可能造成“问得到但打不开”或越权访问争议。
3. 历史文档命名不规范，映射补齐成本较高。
4. 跨子公司链路稳定性不足，影响打开体验。
5. 受控访问设计不当可能泄露内部路径或存在参数枚举风险。

### 19.2 对应处理原则
1. 将 FastGPT 引用字段明确标注为待验证项，不写死。
2. 采用“业务文档权限基线优先”的推荐口径，并列出待确认项。
3. 先支持最小可用映射，再逐步推进规范化。
4. 统一使用桥接系统作为跨网络出口，减少新增链路。
5. 采用 opaque_id + 签名 + 时效 + 服务端鉴权模式。

### 19.3 必须在开发前确认的事项
1. FastGPT 实际稳定返回哪些引用字段。
2. 权限基线具体来源系统或最小配置方案。
3. 飞书身份与内部权限主体的映射方式。
4. 原始 PDF 当前存储位置及管理责任人。
5. 审计日志落点、留存周期、查询权限。

## 20. 实施阶段规划与演进路线

### 20.1 Phase 1：MVP（本期目标）
目标：尽快打通“飞书引用 → 受控打开原始 PDF”最小闭环。

包含：
1. 梳理并验证 FastGPT 可用引用字段。
2. 建立最小映射表。
3. 桥接系统实现引用后处理。
4. 生成受控单文件访问链接。
5. 完成最小权限校验与审计留痕。
6. 支持跨子公司统一桥接访问路径。

### 20.2 Phase 2：稳定化与半自动化
1. 优化映射维护流程，减少人工补录。
2. 完善监控、告警、失败分类。
3. 提升权限源接入质量。
4. 支持历史文档补齐映射。

### 20.3 Phase 3：增强能力
1. 页码级定位。
2. 段落级定位。
3. 多文件类型原文打开。
4. 更精细的分级访问控制。
5. 自动化版本管理与映射修复。

## 21. 验收标准

### 21.1 产品验收标准
1. 文档准确描述当前真实架构与链路，不脱离现状。
2. 明确说明本次核心问题是飞书引用来源补充与真实原始 PDF 受控打开能力。
3. 明确说明该问题是系统补链优化，而非仅改 FastGPT。
4. 明确设计引用标识与原 PDF 的映射机制。
5. 明确桥接系统的后处理职责。
6. 明确单文件受控访问能力及其安全边界。
7. 明确权限一致性与权限基线推荐方案。
8. 明确审计日志最小字段集。
9. 明确跨子公司访问链路约束。
10. 明确映射维护方式与后续演进路线。

### 21.2 边界验收标准
1. 不开放目录浏览。
2. 不暴露服务器真实路径。
3. 不将原 PDF 放到飞书云。
4. 不依赖模型自由生成真实链接。
5. 不在本期引入大规模架构重构。

### 21.3 不确定性表达验收标准
1. 已确认事实、方案假设、待验证项需显式区分。
2. FastGPT 稳定可返回的引用字段必须标注为待验证项。
3. PDF 打开权限“以谁为准”必须给出推荐方案与待确认项。

---

## 22. 结论
本次 PRD 2.0 的核心不是做一个“更花哨的引用链接”，而是在现有企业内部 AI 知识问答系统中，补齐从飞书答案到真实原始 PDF 的可信追溯与受控访问链路。

在不改变当前基础架构的前提下，本方案通过“映射关系 + 桥接后处理 + 单文件受控访问 + 权限基线 + 审计追溯”的组合，形成了一条低风险、工程化、可审计、可演进的补链路径。

若后续待验证项确认顺利，本方案可作为本期落地实施与研发排期的直接依据。
